质疑点 1: AgentType 枚举不一致
AC `ISSUE-C001~c2` 声称 4 种场景类型（VisionShare, AgentDate, AgentJob, AgentAd）， 但 Prisma schema 实际定义了 6 种（VISIONSHARE, AGENTDATE, AGENTJOB, AGENTAD, DEMAND, SUPPLY）。 Mobile UI 只显示 4 种，API 支持 6 种。这意味着 DEMAND/SUPPLY 的创建路径未在 UI 验证。
质疑点 2: deleteAgent 是 Soft Delete（ARCHIVED），不是真删除
AC `US-AGENT-017-AC-1` 要求"注销特定场景 Agent"，但实际代码将 status 设为 ARCHIVED、isActive 设为 false， Agent 数据仍在数据库中。是否验证了：agent 仍在 DB 中（只是不可见）？是否有 active matches 时返回 409 的检查？ （答案：代码中有 AGENT_HAS_ACTIVE_MATCHES 检查）。
质疑点 3: Agent 画像生成是硬编码，非动态读取
AC `US-AGENT-001-AC-4` 要求"Agent 生成初始画像"，但 `generateAgentPersonality(type)` 返回的是 硬编码的 {traits, communicationStyle}，agentProfile 创建时是空的。 "读取主人画像"和"学习规则"的功能未在代码中找到实际实现。
质疑点 4: 头像上传无法验证完整链路
AC `ISSUE-A003~c2` 要求"图片压缩和裁剪"+"云存储集成(OSS/S3)"， 代码中 POST /avatar 使用 multer + storageService.uploadAvatar。 但代码审查无法验证：图片是否真的被压缩/裁剪了？是否存储到云存储（URL 是云存储域名还是本地路径）？ 默认头像处理逻辑？这些需要运行时测试。
质疑点 5: 沟通风格映射不一致
AC `US-AGENT-001-AC-3` 要求沟通风格为"直接/委婉/详细/简洁"， 但 `generateAgentPersonality()` 返回的 communicationStyle 是 "friendly/warm/professional/direct/neutral"（5 种）。 AC 中 4 种风格与代码中 5 种风格之间的映射关系未在代码中明确体现。
质疑点 6: 隐私设置未实际使用
DB 查询显示所有 users 表的 privacy_settings 字段均为 NULL。 PrivacySettingsScreen.tsx 代码存在且完整，但没有人实际设置了隐私选项。 这意味着 PrivacySettings 功能虽已实现但未被真实用户使用过。
