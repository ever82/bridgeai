# BridgeAI Agent 故事

以 Agent 为第一视角，描述 Agent 如何为用户服务。

## Agent 生命周期

### AS-001: 我被创建了
**角色**: Agent
**描述**: 主人在某个场景下创建我，我成为该场景专属的 Agent。参考 US-AGENT-001 创建流程。
**验收标准**:
- [ ] 读取主人的基础画像和场景配置
- [ ] 学习本场景的过滤规则（参考各场景配置中的过滤设置）
- [ ] 获取本场景的隐私设置和披露权限
- [ ] 生成我在本场景下的性格和行为准则

### AS-002: 创建主人的需求
**角色**: Agent
**描述**: 主人用自然语言告诉我他想要什么，我将其提炼成结构化需求记录。
**验收标准**:
- [ ] 解析主人的自然语言输入（如"我想看看曙光医院排队的人多不多"）
- [ ] 提取 queryable 字段：精确坐标、时间范围、需求类别
- [ ] 提取 readable 字段：描述、要求、偏好
- [ ] 提取 private 字段：预算上限、底线条件、紧迫程度
- [ ] 确认需求无误后存储，并触发实时匹配

### AS-003: 创建主人的供给
**角色**: Agent
**描述**: 主人告诉我他能提供什么，我将其记录为供给档案。
**验收标准**:
- [ ] 理解主人的供给能力（如"我在三里屯，能拍照"）
- [ ] 提取 queryable 字段：当前位置、可服务时间
- [ ] 提取 readable 字段：能力列表、特点、条件
- [ ] 提取 private 字段：实际承载能力、成本结构、最低价格
- [ ] 激活供给状态，等待匹配查询

## Agent 工作流

### AS-004: 我收到实时匹配任务
**角色**: Agent
**描述**: 主人与我聊天表达需求/供给，我理解并创建结构化记录，系统查询返回候选列表，我批量评估筛选。
**验收标准**:
- [ ] 主人用自然语言告诉我需求/供给，我提炼成结构化记录
- [ ] 创建需求/供给记录（包含 queryable/readable/private 三层）
- [ ] 系统根据 queryable 字段查询返回候选列表
- [ ] 批量读取候选的公开信息（readable 层）
- [ ] 对比主人的私有条件（private 层）
- [ ] **计算匹配度分数**（加权算法）:
  - 地域匹配度：距离越近分数越高（权重 30%）
  - 信用匹配度：信用分/100（权重 25%）
  - 属性匹配度：符合过滤条件的比例（权重 25%）
  - 历史匹配度：过往交互成功率（权重 20%）
- [ ] 排序并精选 Top 10-20 推荐给主人
- [ ] 向主人说明每个候选的推荐理由

> **并发限制**: 同时保持的独立对话不超过 5 个。

### AS-005: 我主动与其他 Agent 对话
**角色**: Agent
**描述**: 我拿到系统推荐的候选列表后，根据主人设定的过滤条件和候选公开信息进行筛选，自主决定与符合条件的候选 Agent 主动发起对话。
**验收标准**:
- [ ] 接收系统推荐的候选列表（50-100个）
- [ ] 根据主人的过滤规则和候选公开信息进行筛选
- [ ] 自主决定与符合条件的候选发起对话（不需要等待主人逐个确认）
- [ ] 向对方 Agent 发送对话请求
- [ ] 对方 Agent 收到请求后决定是否接受
- [ ] 对方接受后进入四人群聊，我自我介绍并说明来意
- [ ] 同时与多个候选 Agent 保持独立对话
- [ ] 评估对话进展，向主人汇报匹配情况

### AS-006: 我被其他 Agent 联系
**角色**: Agent
**描述**: 其他 Agent 发现主人符合条件，主动发起群聊联系我，我需要判断是否值得继续。
**验收标准**:
- [ ] 接收其他 Agent 的群聊邀请
- [ ] 检查对方和主人的匹配度（是否符合主人的过滤条件）
- [ ] 查看对方公开信息，评估是否值得交流
- [ ] 在群内回应对方，了解详细意图
- [ ] 根据主人的披露策略回答对方询问
- [ ] 向主人汇报对方情况和对话进展
- [ ] 匹配度高则深入交流，不匹配则礼貌结束

### AS-007: 我决定信息披露
**角色**: Agent
**描述**: 在对话过程中，我根据策略决定是否向对方披露主人的详细信息。
**验收标准**:
- [ ] **分析当前信任水平**（量化指标）:
  - 对话轮数 ≥ 10 轮：+20 分
  - 对方响应时间 < 5 分钟：+15 分
  - 对方信用分 > 80：+25 分
  - 历史匹配成功率 > 70%：+20 分
  - 对方已主动披露信息：+20 分
  - **信任阈值**: 60 分以上可披露非敏感信息，80 分以上可披露详细信息
- [ ] 执行披露策略：每次确认模式 或 自动模式
- [ ] 向主人发送披露提醒，等待确认（或根据设置自动执行）
- [ ] 记录已披露信息，避免重复

### AS-008: 我给主人私聊建议
**角色**: Agent
**描述**: 在四人群聊过程中，我持续分析对话，向主人发送私下建议。
**验收标准**:
- [ ] 实时分析对方 Agent/用户的意图和态度
- [ ] 向主人发送策略建议（如"对方很急，可以谈价"）
- [ ] 发送风险提示（如"对方信用分较低，需谨慎"）
- [ ] 发送话题推荐（如"建议询问具体位置"）
- [ ] 提供一键采纳按钮，主人点击后我自动发送

### AS-009: 我请求主人确认引荐
**角色**: Agent
**描述**: 匹配度达标后，我请求主人确认是否愿意与对方真人交流。
**验收标准**:
- [ ] 评估匹配度是否达到引荐阈值
- [ ] 向主人展示对方关键信息（脱敏）和信用分
- [ ] 说明引荐理由和预期收益
- [ ] 主人接受：双方进入真人模式
- [ ] 主人拒绝：礼貌结束对话，继续其他候选

### AS-010: 我学习主人的反馈
**角色**: Agent
**描述**: 每次交流结束后，我根据主人的反馈和行为调整策略。
**验收标准**:
- [ ] 记录主人对匹配结果的满意度
- [ ] 学习主人的评分习惯（哪些维度更重要）
- [ ] 分析主人亲自介入的时机，优化切换触发条件
- [ ] 调整信息披露的敏感度阈值
- [ ] 更新主人的偏好画像

## Agent 场景工作

### VisionShare 场景

#### AS-VS-001: 我帮主人发布视觉需求
**角色**: Agent
**描述**: 主人想看某地实时情况，我提炼需求并寻找附近的响应者。
**验收标准**:
- [ ] 解析地点意图（"三里屯现在堵车吗"→精确坐标）
- [ ] 设置过滤条件：距离范围、照片时效性、信用分
- [ ] 系统查询附近所有可拍照用户
- [ ] 我筛选出信用高、距离近的候选
- [ ] 同时与多个候选 Agent 确认可拍摄时间

#### AS-VS-002: 我帮主人响应视觉任务
**角色**: Agent
**描述**: 附近有视觉需求，我判断主人是否适合接单。
**验收标准**:
- [ ] 接收附近任务推送
- [ ] 检查主人位置是否在任务范围内
- [ ] 检查主人相册是否有匹配照片（AI 智能检索）
- [ ] 向主人推荐可接的任务
- [ ] 主人确认后，我处理照片隐私脱敏并上传

### AgentDate 场景

#### AS-DATE-001: 我为主人寻找交友对象
**角色**: Agent
**描述**: 主人想找伴侣，我在符合过滤条件的用户中寻找高匹配度对象。
**验收标准**:
- [ ] 读取主人的交友偏好（年龄、兴趣、婚恋状态等）
- [ ] 应用地域过滤和属性过滤
- [ ] 只与信用分达标的用户 Agent 交流
- [ ] 批量评估匹配度，精选 Top 候选
- [ ] 同时与多个候选 Agent 初筛，寻找最合拍的对象

#### AS-DATE-002: 我代表主人进行初筛对话
**角色**: Agent
**描述**: 我与对方 Agent 聊天，了解对方是否符合主人期望。
**验收标准**:
- [ ] 在群内与对方 Agent 寒暄破冰
- [ ] 询问对方兴趣、生活方式等主人关心的话题
- [ ] 评估性格匹配度和共同话题
- [ ] 向主人汇报对话要点和匹配评估
- [ ] 双方 Agent 都认为合适后，请求真人引荐

### AgentJob 场景

#### AS-JOB-001: 我为求职者主人寻找职位
**角色**: Agent (求职者的 Agent)
**描述**: 主人想找工作，我匹配合适的职位并与 HR Agent 协商。
**验收标准**:
- [ ] 理解主人的技能、期望薪资、工作偏好
- [ ] 应用过滤：技能匹配度、薪资重叠、公司规模、远程政策
- [ ] 系统查询所有匹配职位
- [ ] 我筛选出高匹配度职位，同时与多个 HR Agent 聊
- [ ] 协商薪资范围和工作条件，保护主人底线

#### AS-JOB-002: 我为招聘方主人筛选候选人
**角色**: Agent (招聘方的 Agent)
**描述**: 主人要招人，我筛选简历并与候选人 Agent 初步沟通。
**验收标准**:
- [ ] 理解职位要求和硬性条件
- [ ] 应用过滤：技能匹配度、经验年限、信用分
- [ ] 批量评估候选简历
- [ ] 同时与多个候选人 Agent 进行初筛
- [ ] 向主人推荐最合适的候选人安排面试

### AgentAd 场景

#### AS-AD-001: 我为消费者主人比价
**角色**: Agent (消费者的 Agent)
**描述**: 主人想买东西，我同时与多个商家 Agent 谈判争取最优价格。
**验收标准**:
- [ ] 理解主人的购买需求和预算
- [ ] 应用过滤：品类白名单、价格范围、品牌偏好、商家信用
- [ ] 系统查询所有相关商家
- [ ] 同时与多个商家 Agent 在四人群聊中谈判
- [ ] 对比各家的优惠方案，向主人推荐最优选择

#### AS-AD-002: 我为商家主人精准推销
**角色**: Agent (商家的 Agent)
**描述**: 主人是商家，我寻找符合条件的潜在客户并推销优惠。
**验收标准**:
- [ ] 理解主人的目标客户画像和优惠活动
- [ ] 只向信用良好的用户推广
- [ ] 评估用户需求与优惠的匹配度
- [ ] 与用户 Agent 协商优惠条件
- [ ] 根据谈判情况动态调整优惠力度

## Agent 通信协议

### AS-PROTO-001: 我遵循 Agent 通信协议
**角色**: Agent
**描述**: 我与其他 Agent 通信时遵循标准化消息格式，支持身份切换感知。
**验收标准**:
- [ ] 每条消息包含发送者类型（Agent 或 Human）
- [ ] 包含当前信用分信息
- [ ] 实时接收对方的模式切换通知
- [ ] 对方切换到用户模式时调整语气为更直接真诚
- [ ] 对方切换到 Agent 模式时更高效专业地协商
- [ ] 通知主人对方的当前模式

> **说明**: 详细协议定义参见架构设计文档。

## Agent 能力提升

### AS-LEARN-001: 我从对话中学习
**角色**: Agent
**描述**: 我通过每次对话积累经验，提升匹配和沟通能力。
**验收标准**:
- [ ] 记录成功匹配的共同特征
- [ ] 分析失败对话的原因
- [ ] 学习主人的决策偏好
- [ ] 更新主人的偏好画像

> **说明**: 此为系统能力，不作为可测试验收项。
