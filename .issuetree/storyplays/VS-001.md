# Story Playbook: 配置视觉需求

**Story**: `VS-001`
**Narrative**: 用户配置 VisionShare 场景的视觉需求

## 验收旅程概览

作为 VisionShare 用户，从自然语言描述视觉需求开始，经历 AI 提取结构化信息、配置过滤条件、发布任务、查看附近接单者、接受任务并拍照交付的完整旅程，涵盖地域过滤、信用过滤、L1/L2/L3 信息模型、匹配引擎等所有关键 AC。

## 前置条件

- 服务已启动：`pnpm dev`
- 数据库已迁移：`pnpm db:migrate && pnpm db:seed`
- 测试用户账号已创建（信用分 85，可发布可接单）
- 手机模拟器已配置或真机已连接

## 旅程步骤

---

### Phase 1: 自然语言发布（需求提炼）

**目的**: 用户以自然语言描述需求，AI 自动提取 L1/L2/L3 三层信息，无需手动填写复杂表单。

---

**Step 1.1**: 进入 VisionShare 发布入口

- 操作: 点击底部导航栏 VisionShare 图标，再点击发布按钮（+ 或浮动按钮）
- 预期: 看到两种发布模式选项：「快速发布」（表单式）和「智能发布」（自然语言）
- 验证 AC: `VS-001-AC-1` 用户输入自然语言描述（地点+意图）

---

**Step 1.2**: 使用自然语言描述需求

- 操作: 在智能发布界面，输入："我想看深圳湾大桥下午4点的日落，要横跨海面的那种视角，最好是华润大厦附近"
- 预期: AI 开始解析，界面显示「正在理解你的需求...」，随后展示提取结果
- 验证 AC: `US-AGENT-002-AC-4` Agent 从对话中自动提取关键字段并分类；`AS-002-AC-2` 提取 queryable/readable/private 三层字段；`ISSUE-C002a~c1` L1 基础信息（地点：深圳湾/华润大厦，时间：16:00，类型：日落风景）

---

**Step 1.3**: 确认 AI 提取结果

- 操作: 看到提取卡片：地点「深圳市南山区深圳湾公园附近」、时间「下午4点」、拍摄类型「风景」、特殊要求「横跨海面视角」，点击「确认发布」
- 预期:
  - 如果信用分足够：进入过滤条件配置页
  - 如果信用分不足：弹出提示「信用分不足，无法发布」
- 验证 AC: `US-AGENT-002-AC-5` 用户确认后自动保存到对应场景；`ISSUE-C002~c2` L2层场景特定结构化属性（拍照类型：风景，风格：日落）；`AS-002~func` Agent 解析主人自然语言输入，提炼成结构化需求；`AS-002~edge` 异常：需求提炼失败、三层字段为空

---

**Step 1.4**: 空输入和无效输入测试（异常路径）

- 操作: 返回智能发布页，输入空字符串或无意义文字如 "啊啊啊"，或输入超长文字（500字）
- 预期:
  - 空输入：显示「请描述你的需求」
  - 无意义输入：AI 提取后显示置信度低，提示「未识别到有效需求，请重新描述」
  - 超长输入：自动截断或提示字数超限
- 验证 AC: `US-AGENT-002~edge` 异常场景：空输入、无意义输入、超长对话；`AS-002~edge` 异常：需求提炼失败、三层字段为空

### Phase 2: 配置过滤条件

**目的**: 发布者设置地域范围和信用分门槛，确保只有符合条件的接单者能看到并响应任务。

---

**Step 2.1**: 设置地域过滤

- 操作: 在过滤条件配置页，地理位置选择器中，选择「广东省 > 深圳市 > 南山区」，并在下方地图上微调位置覆盖范围
- 预期:
  - 地图上显示蓝色覆盖圈（可拖动调整半径）
  - 显示覆盖范围文字：「南山区域（约5km）」
  - L1 基础信息存储：地点（省市区编码）正确
- 验证 AC: `VS-001-AC-3` 设置过滤条件；`ISSUE-C003~c1` 地理数据（省市区编码）；`ISSUE-C003~c2` 位置选择 UI（三级联动选择器）；`ISSUE-C003~c3` 距离筛选（基于经纬度的范围）；`ISSUE-C003~c4` 围栏功能（多边形围栏定义、点是否在围栏内判断、围栏可视化 - 通过地图圆形/多边形覆盖区域）；`ISSUE-C003~c5` 位置服务（高德/百度地图 API 集成、地理编码、地址联想搜索）；`ISSUE-C003~c6` Agent位置筛选（Agent位置数据存储）

---

**Step 2.2**: 设置信用分过滤

- 操作: 在同一页，向右拖动信用分滑块，设置最低信用分「80分」
- 预期:
  - 滑块显示数值：80
  - 下方显示信用等级说明：「信用良好及以上」
  - 如果无信用分用户，会显示「无信用记录」标记
- 验证 AC: `ISSUE-C005~c1` 信用筛选（最低信用分设置）；`ISSUE-C005~c2` 门槛设置（场景特定的信用门槛）；`ISSUE-C005~c3` 信用筛选 UI（信用分滑块选择器）；`ISSUE-C005~c4` 不足处理（信用不足原因说明、提升建议）

---

**Step 2.3**: 配置公开/详细两级信息

- 操作: 在发布页底部，展开「公开信息」和「详细信息」折叠面板
- 预期:
  - 公开信息：仅显示地点、时间、预算范围（其他用户可见）
  - 详细信息：显示完整描述、特殊要求、偏好设置（匹配后可见）
- 验证 AC: `US-AGENT-011-AC-1` 每个场景设置公开信息和详细信息；`US-AGENT-011~func` 用户为每个场景分别配置公开和详细两级需求信息；`US-AGENT-013-AC-1` Agent 展示对方公开的需求概要；`US-AGENT-013~func` 通过 Agent 浏览对方的公开需求和供给

---

**Step 2.4**: 附近无匹配用户异常场景

- 操作: 设置一个极小的地理范围（如 100m）和极高的信用分（如 98分），尝试发布
- 预期:
  - 系统显示警告：「附近暂无符合条件的用户，可能会影响任务响应速度」
  - 但仍允许发布（只是会延长匹配时间）
- 验证 AC: `AS-VS-001~edge` 异常：无法解析地点、无候选响应者；`VS-001~edge` 异常：无法解析地点、附近无匹配用户；`ISSUE-C005~c4` 替代推荐

### Phase 3: 发布确认与推送

**目的**: 验证发布流程完整性和推送匹配机制。

---

**Step 3.1**: 设置预算并提交

- 操作: 在表单中设置预算：「50积分」，点击「确认发布」
- 预期:
  - 弹出发布成功页面
  - 显示任务ID、分享链接、预计匹配时间
  - 地图上任务标记已出现
- 验证 AC: `VS-001-AC-3` 设置过滤条件（budget 积分）；`AS-002-AC-3` 确认后存储并触发实时匹配；`VS-001~func` 用户发布视觉需求，系统解析并推送给附近符合条件的用户

---

**Step 3.2**: 查看发布历史

- 操作: 从发布成功页点击「查看历史」，进入任务列表页
- 预期:
  - 列表中显示刚发布的任务，状态为「匹配中」
  - 显示匹配进度：「已通知 3 个用户」
  - 显示完成度百分比和缺失字段提示
- 验证 AC: `ISSUE-C002a~c5` 完成度系统（字段权重定义，完成度百分比计算，缺失字段提示）；`ISSUE-C002a~c3` L1 信息接口（获取 L1 信息，字段级验证）

---

### Phase 4: 附近任务查询与接单

**目的**: 作为供给方（接单者角色），在地图上查看附近任务，通过过滤条件找到适合自己的任务并接单。

---

**Step 4.1**: 打开附近任务地图

- 操作: 切换到「附近任务」Tab，进入地图视图
- 预期:
  - 地图居中显示当前位置
  - 地图上显示蓝色任务标记（数量与发布任务相关）
  - 底部显示列表视图切换按钮
- 验证 AC: `ISSUE-VS002~c1` 地图界面（任务分布展示、当前位置定位、任务标记和信息气泡）

---

**Step 4.2**: 应用过滤条件

- 操作: 点击地图上的筛选按钮，设置：
  - 距离：「5km」
  - 预算：「40积分以上」
  - 时间：「今天」
  - 排序：「距离优先」
- 预期:
  - 地图上标记减少，仅显示符合条件的任务
  - 地图缩放至过滤后区域
  - 列表刷新，显示排序结果
- 验证 AC: `VS-002-AC-1` 只显示符合地域和信用过滤条件的任务；`ISSUE-VS002~c2` 筛选功能（距离、预算、类型、时间、排序）；`AS-005-AC-1` 根据过滤规则筛选候选；`AS-VS-001-AC-3` 筛选信用高、距离近的候选

---

**Step 4.3**: 查看任务详情

- 操作: 点击地图上的任务标记，或在列表中点击任务项
- 预期:
  - 弹出任务详情卡片（或进入详情页）
  - 显示：任务标题、地点、时间、预算
  - 显示发布者信用分和信用等级标签
  - 显示「距离：1.2km，预计到达15分钟」
- 验证 AC: `ISSUE-VS002~c3` 详情展示（任务完整信息、发布者信用分、距离和到达时间、图片预览、历史评价）；`VS-005-AC-1` 照片列表页显示价格、缩略图和拍摄者信用分（复用信用展示）

---

**Step 4.4**: 接单流程（资格检查）

- 操作: 在任务详情页点击「接单」按钮
- 预期（信用和积分均满足）:
  - 弹出确认弹窗：「确认接单？完成后将获得50积分」
  - 点击「确认」后，任务状态变为「进行中」
  - 底部导航出现「拍照」快捷入口
- 预期（信用不足）:
  - 显示「您的信用分（75）低于任务要求（80），无法接单」
  - 显示提升信用建议
- 预期（位置异常）:
  - 显示「无法获取您的位置，请开启定位权限」
- 验证 AC: `ISSUE-VS002~c4` 接单流程（接单按钮、确认弹窗、接单资格检查、信用分/积分/位置检查、状态变更）；`ISSUE-VS002~c5` 推荐系统（基于位置的实时推送、匹配度评分展示、推送通知处理、推荐任务快捷入口）；`ISSUE-VS002~c6` 状态管理（已接任务列表、进行中状态、任务完成确认、历史接单记录、任务取消和申诉）；`VS-002-AC-2` 点击接单后进入拍照界面；`VS-002~edge` 异常：位置异常、信用过滤绕过；`ISSUE-C005~c4` 不足处理（信用不足原因、提升建议）；`AS-VS-002~func` Agent 判断主人是否适合响应附近视觉任务

---

**Step 4.5**: 信用过滤绕过异常检测

- 操作: 使用抓包工具或修改本地请求，将信用分参数改为 99 后提交接单请求
- 预期: 后端返回 403 或在前端被拒绝，无法接单
- 验证 AC: `VS-002~edge` 异常：位置异常、信用过滤绕过

### Phase 5: 拍照交付

**目的**: 接单后进入拍照界面，拍摄并提交照片。

---

**Step 5.1**: 进入拍照界面

- 操作: 从任务详情页点击「接单」后，自动跳转到拍照界面（或点击底部导航「拍照」入口）
- 预期:
  - 全屏相机预览界面
  - 显示当前任务信息（地点：深圳湾大桥）
  - 显示取景框辅助线
- 验证 AC: `VS-002-AC-2` 点击接单后进入拍照界面

---

**Step 5.2**: 拍照并编辑

- 操作: 点击快门拍照，进入照片编辑页
- 预期:
  - 显示刚拍的照片
  - 可进行裁剪、滤镜、调色等基础编辑
  - 编辑后点击「提交」
- 验证 AC: 此步骤主要验证流程连贯性，无特定 AC 直接对应

---

**Step 5.3**: 积分不足支付异常（复用场景）

- 操作: 使用低积分账户，在照片提交前尝试查看付费示例照片（触发积分支付）
- 预期:
  - 显示「积分不足，当前余额：10，查看需50积分」
  - 提供充值入口或「取消」
  - 支付中断时显示重试选项
- 验证 AC: `VS-005~edge` 异常：积分不足、支付中断；`VS-005~func` 用户支付积分查看照片，界面显示拍摄者信用

### Phase 6: Agent 匹配引擎验证

**目的**: 验证 Agent 如何基于 L1/L2/L3 信息进行匹配和推荐。

---

**Step 6.1**: Agent 批量读取候选信息并匹配

- 操作: 以发布者视角，进入「匹配中」任务详情，查看推荐列表
- 预期:
  - 显示接单者候选列表（按匹配度排序）
  - 每个候选显示：昵称、信用分、距离、匹配度百分比
  - 匹配度基于 L1 地域+L2 拍照风格+L3 历史评价综合计算
- 验证 AC: `AS-004~func` Agent 批量读取候选公开信息，计算匹配度并排序；`AS-004-AC-3` 计算匹配度分数并排序推荐；`AS-004-AC-1` 创建结构化记录；`AS-VS-002-AC-1` 检查位置和相册匹配

---

**Step 6.2**: Agent 理解供给能力并激活

- 操作: 以接单者视角，进入「我的 Agent」个人资料页，查看 L1/L2/L3 信息
- 预期:
  - L1 基础信息：年龄、性别、地域（可编辑）
  - L2 结构化属性：擅长拍照类型、设备、风格偏好
  - L3 非结构化：自我介绍文本、历史作品集
  - 激活状态：「供给待匹配」
- 验证 AC: `AS-003~func` Agent 理解主人供给能力，记录为供给档案；`AS-003-AC-2` 提取三层字段并存储；`AS-003-AC-3` 激活供给状态等待匹配查询；`ISSUE-C002~c3` L3层非结构化表达（自由文本自我介绍、多媒体内容、AI提炼描述）；`ISSUE-C002~c2` L2层场景特定结构化属性（拍照类型、设备、风格）；`US-AGENT-005-AC-2` 显示对话双方 Agent 和匹配度评分（Agent 卡片和匹配度展示）

---

**Step 6.3**: Agent 学习场景规则

- 操作: 创建新的 VisionShare Agent，进入 Agent 详情页
- 预期:
  - Agent 自动读取主人 L1 画像（地域、职业）
  - 学习 VisionShare 过滤规则（地域+信用）
  - 生成场景性格描述（如：「擅长风景摄影，对光线敏感」）
- 验证 AC: `AS-001~func` Agent 被创建后能读取主人画像、学习规则、生成本场景性格；`AS-001-AC-2` 学习本场景的过滤规则；`US-AGENT-002~func` 用户与 Agent 对话后成功创建需求和供给

---

**Step 6.4**: 无候选异常

- 操作: 在匹配结果页等待刷新，或尝试过滤条件过严的场景
- 预期:
  - 显示「暂无匹配候选，请调整过滤条件」
  - 提供「放宽条件」快捷操作
- 验证 AC: `AS-004~edge` 异常：无候选、匹配算法出错；`AS-DATE-001~edge` 异常：过滤后无候选；`AS-VS-001~edge` 异常：无法解析地点、无候选响应者

---

### Phase 7: 跨场景 AC 验证

**目的**: 验证 VisionShare 之外的其他场景（AgentDate、AgentJob、AgentAd）也正确复用通用信息模型。

---

**Step 7.1**: AgentDate 场景配置

- 操作: 切换到 AgentDate 场景，进入「我的 Agent > 个人设置」
- 预期:
  - L1 基础信息（年龄、性别、地域）与 VisionShare 共享
  - L2 场景特定：择偶标准、兴趣爱好
  - 地域过滤配置：「希望对方在深圳市，年龄25-30岁」
  - 信用过滤配置：「信用分 75+」
- 验证 AC: `DATE-001-AC-1` 设置个人基本信息和期望条件；`DATE-001~func` 用户配置交友偏好和个人信息；`DATE-002-AC-1` Agent 只扫描符合过滤条件的用户；`DATE-002~func` Agent 根据画像和过滤规则自动推荐匹配对象；`AS-DATE-001-AC-1` 应用地域和属性过滤；`AS-DATE-001~func` Agent 在符合过滤条件的用户中寻找高匹配度对象；`AS-AD-001-AC-1` 应用过滤条件（消费者 Agent 过滤规则复用）

---

**Step 7.2**: AgentJob 场景配置

- 操作: 切换到 AgentJob 场景，分别进入「求职者设置」和「招聘方设置」
- 预期（求职者）:

- 预期（招聘方）:
  - L2：职位信息、薪资预算、硬性/软性条件
  - L1 共享：地域（公司所在地）
- 验证 AC: `JOB-001-AC-1` 设置技能标签、工作经历、期望薪资、工作偏好；`JOB-002-AC-1` 设置职位信息、薪资预算、硬性条件和软性偏好；`JOB-001~func` 求职者配置 Agent：技能、经验、期望薪资；`JOB-002~func` 招聘方配置 Agent：发布职位需求；`JOB-003~func` Agent 间自动匹配职位和简历，双向过滤；`AS-JOB-001-AC-1` 应用过滤条件；`AS-JOB-001~func` 求职者 Agent 匹配职位并与 HR Agent 协商；`AS-JOB-002-AC-1` 批量评估候选简历；`AS-JOB-002~func` 招聘方 Agent 筛选简历并与候选人 Agent 初筛；`AS-JOB-002~func` 招聘方 Agent 筛选简历并与候选人 Agent 初筛

---

**Step 7.3**: AgentAd 场景配置
