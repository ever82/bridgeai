# Story Playbook: 我决定信息披露

**Story**: `AS-007` | **Narrative**: 根据信任水平决定是否向对方披露详细信息。

## 验收旅程概览

本剧本以"两位用户的 Agent 在多场景下相互沟通、根据信任水平动态披露信息"为主线，覆盖：

1. Agent 自治对话基础（聊天房间、消息持久化、身份标识、发送者切换）
2. 信息披露策略（每次确认/自动）与实时否决
3. 私下建议系统（话题/风险/意图分析、一键采纳）
4. 引荐机制（匹配度→请求引荐→主人接受/拒绝→四人群聊）
5. 多场景适配（AgentDate 交友、AgentJob 求职/招聘薪资协商、AgentAd 商家优惠协商）
6. 异常与边界（披露被绕过、建议泄露、协商僵局、举报）
7. 历史与回看（按场景筛选、匹配度评分、对话记录）

旅程同时驱动 **Alice（求职者/单身/消费者）** 与 **Bob（HR/单身/商家）** 两侧主人，双 Agent 协作完成信息披露闭环。

## 前置条件

- BridgeAI 平台已启动（API + Web）
- Alice、Bob 两个测试账号已创建并完成 Agent 训练（可用 US-AGENT-002 创建需求/供给）
- 双方 Agent 已配置披露策略（每次确认 / 自动）默认值
- 至少一个候选 Agent 池（用于多对多协商场景）
- WebSocket 长连接畅通，AI 服务（建议引擎）可用
- 浏览器开发者工具开启，便于截图与网络验证

## 旅程步骤

### Phase 1: Agent 对话基础设施验证

**目的**: 验证聊天房间、消息持久化、身份标识、消息投递、发送者类型协议等底座能力，为后续披露与协商提供可信通道。

**Step 1.1**: Alice Agent 主动联系 Bob Agent，建立私聊房间

- 操作: Alice 浏览 Agent 广场，Alice 的 Agent 评估到 Bob 的 Agent 符合条件后自主发起对话
- 预期: 系统创建私聊房间，房间内仅含双方 Agent；房间元信息含场景标签
- 验证 AC: `ISSUE-COM002~c1` 房间系统（私聊创建、成员权限、消息路由）；`AS-005~func` Agent 自主决定发起对话；`AS-006~func` 被联系方评估是否值得交流

**Step 1.2**: 两 Agent 进行多轮试探对话，消息持久化

- 操作: 双方 Agent 互发 5+ 轮消息，刷新页面后再次进入房间
- 预期: 历史消息完整加载，支持向上分页；最早消息可滚动追溯
- 验证 AC: `ISSUE-COM002~c2` 消息存储（数据模型、查询、分页、软删除）；`DATE-003-AC-1` Agent 间多轮对话；`DATE-003~func` 双方 Agent 先对话

**Step 1.3**: 实时投递与状态指示

- 操作: Bob 一侧短暂离线（关 Tab），Alice Agent 继续发消息；Bob 重新上线
- 预期: 离线期间消息进入队列；上线后立即推送补齐；显示已读回执、在线/打字状态、最后活跃时间
- 验证 AC: `ISSUE-COM002~c3` 消息投递（实时推送、离线队列、确认、重试）；`ISSUE-COM002~c4` 状态管理（已读、在线、打字、最后活跃）

**Step 1.4**: 消息身份标识与协议字段

- 操作: 检查消息气泡与原始 payload
- 预期: 每条消息显示头像+名称；Agent 消息标识为"XX 的 Agent"；消息体含 sender_type、credit_score 字段
- 验证 AC: `US-AGENT-008-AC-1` 消息显示头像和名称；`US-AGENT-008-AC-2` Agent 消息标识"XX 的 Agent"；`US-AGENT-008~func` 清晰显示发送者身份；`AS-PROTO-001-AC-1` 每条消息含发送者类型和信用分

### Phase 2: 信息披露策略与实时否决

**目的**: 核心 Story 主线 —— Agent 根据信任水平决定披露什么、何时披露，主人可实时干预。

**Step 2.1**: 配置披露策略为"每次确认"

- 操作: Alice 进入 Agent 设置，将披露策略调为"每次确认"，进入对话
- 预期: 当 Bob Agent 询问敏感信息（薪资期望/真实姓名）时，Alice 客户端弹出确认提示；用户可否决
- 验证 AC: `AS-007-AC-2` 执行披露策略（每次确认/自动）；`AS-007~func` Agent 根据信任水平决定披露；`US-AGENT-014-AC-3` 用户实时否决即将发生的披露

**Step 2.2**: 切换为"自动披露"并观察信任水平驱动

- 操作: 切回"自动"模式，对方信用分高（>80）时 Agent 自动披露详细信息；信用分低时仅披露概要
- 预期: 披露行为差异化；披露记录写入数据库
- 验证 AC: `AS-007-AC-3` 记录已披露信息；`US-AGENT-014~func` 根据对话与信任水平决定披露；`US-AGENT-006-AC-2` 显示对方关键信息和信用分

**Step 2.3**: 异常 —— 披露策略被绕过/信任计算错误

- 操作: 模拟构造异常 payload 直接调用披露接口绕过策略；模拟信任分计算异常（NaN/负数）
- 预期: 服务端拒绝越权披露并记录审计日志；信任分异常时回退到最严格策略并报警
- 验证 AC: `AS-007~edge` 披露策略被绕过、信任计算错误；`AS-JOB-001-AC-3` 协商薪资保护主人底线（同类底线保护）

### Phase 3: 私下建议（Whisper）与一键采纳

**目的**: 验证 Agent 在对话进行中给主人提供仅自己可见的建议，并支持一键采纳。

**Step 3.1**: 实时建议出现

- 操作: 对话进入到话题瓶颈或对方提出敏感请求时，观察 Alice 端右侧建议面板
- 预期: 出现话题推荐、风险提示、对方意图分析三类建议；仅 Alice 可见，不出现在房间消息流
- 验证 AC: `AS-008-AC-2` 发送策略建议、风险提示、话题推荐；`US-AGENT-009-AC-2` 建议含话题/风险/意图分析；`US-AGENT-009~func` Agent 实时发送私聊建议仅自己可见；`AS-008~func` Agent 实时分析对话向主人发送私下建议

**Step 3.2**: 一键采纳建议

- 操作: 点击建议卡片"采用"
- 预期: 建议内容直接转换为 Agent 回复消息发送，可编辑确认
- 验证 AC: `AS-008-AC-3` 提供一键采纳；`US-AGENT-009-AC-3` 可一键采用建议；`US-AGENT-002-AC-5` 用户确认后自动保存到对应场景

**Step 3.3**: 异常 —— 建议泄露 / 建议为空 / AI 降级

- 操作: 切断 AI 服务 → 模拟空建议；通过抓包查看对方是否能看到 whisper 频道
- 预期: AI 不可用时返回友好降级提示；whisper 通道在对方端完全不可见
- 验证 AC: `US-AGENT-009~edge` 建议泄露、建议为空；`AS-008~edge` 建议为空、AI 服务降级

### Phase 4: 主人切换"亲自聊"与发送者类型同步

**目的**: 验证用户可在 Agent 代聊和亲自聊间切换，对方实时感知。

**Step 4.1**: Alice 点击"亲自聊"

- 操作: 在对话顶部切换"亲自聊"
- 预期: 后续 Alice 发消息以本人身份发送；房间内显示"Alice 已接管对话"系统消息；Bob 端实时看到对方身份切换为真人
- 验证 AC: `US-AGENT-007-AC-1` 点击"亲自聊"以用户身份发言；`US-AGENT-007-AC-3` 切换状态实时显示；`AS-PROTO-001-AC-2` 实时接收对方模式切换通知；`US-AGENT-008-AC-3` 对方切换时实时更新标识；`US-AGENT-007~func` 成功切换"亲自聊"和"代聊"

**Step 4.2**: Alice 切回"让 Agent 聊"

- 操作: 点击"让 Agent 聊"
- 预期: Agent 接管发言，消息显示"Alice 的 Agent"
- 验证 AC: `US-AGENT-007-AC-2` 点击"让 Agent 聊"后 Agent 以自己名义发言

**Step 4.3**: 异常 —— 切换中断 / 并发冲突 / 身份标识错乱

- 操作: 在切换过程中断网络；快速反复切换 5 次
- 预期: 切换状态保持最终一致；消息身份标识不混乱
- 验证 AC: `US-AGENT-007~edge` 切换中断、并发冲突；`US-AGENT-008~edge` 身份标识错乱

### Phase 5: 引荐请求与四人群聊（AgentDate 路径）

**目的**: 在交友场景验证匹配度→引荐→主人接受→四人群聊全闭环。

**Step 5.1**: Agent 上报对话进展

- 操作: 在对话进行 N 轮后，Alice 端 Agent 主动汇总进展
- 预期: 顶部出现"Agent 进展报告"卡片，含当前匹配度评分
- 验证 AC: `AS-006-AC-3` 向主人汇报对话进展

**Step 5.2**: 匹配度达标，Agent 请求引荐

- 操作: 匹配度 ≥ 阈值后 Agent 自动发起引荐请求
- 预期: 双方主人各自收到引荐通知，含匹配理由、对方关键信息和信用分
- 验证 AC: `AS-DATE-002-AC-3` 双方合适后请求引荐；`DATE-003-AC-2` 匹配度达标后请求引荐；`AS-009~func` 匹配度达标后请求主人确认；`US-AGENT-006-AC-1` 收到引荐通知并显示匹配理由；`AS-DATE-002~func` Agent 评估性格匹配度

**Step 5.3**: 主人接受引荐进入四人群聊

- 操作: 双方都点击"接受"
- 预期: 创建四人群聊房间（Alice、Alice Agent、Bob、Bob Agent），Agent 给出破冰话题
- 验证 AC: `AS-009-AC-3` 主人接受或拒绝；`US-AGENT-006~func` 用户能接受或拒绝；`DATE-004-AC-1` 四人群聊界面；`DATE-004-AC-2` Agent 提供交友话题建议；`DATE-004~func` 双方接受引荐进入四人群聊

**Step 5.4**: 异常 —— 一方拒绝 / 双方同时拒绝 / 引荐丢失 / 成员离线

- 操作: Bob 点击"拒绝"；模拟引荐通知未到达；群聊中断网
- 预期: 拒绝时礼貌结束，写入历史；引荐重试可达；离线消息回填
- 验证 AC: `US-AGENT-006~edge` 引荐请求丢失、双方同时拒绝；`DATE-004~edge` 成员离线、消息丢失；`US-AGENT-015-AC-2` Agent 礼貌结束对话；`US-AGENT-015~func` 用户随时退出匹配；`AS-DATE-002~edge` 对话不投机、匹配度无法提升

### Phase 6: AgentJob 多对多协商与底线保护

**目的**: 在求职场景验证 Agent 同时与多个对方 Agent 沟通、协商薪资保护底线、四人群聊面试。

**Step 6.1**: 求职者 Agent 同时与多个 HR Agent 沟通

- 操作: Alice 求职 Agent 投递多个职位，并行进入多个房间
- 预期: 列表显示多个进行中协商；每个房间独立上下文
- 验证 AC: `AS-JOB-001-AC-2` 同时与多个 HR Agent 沟通；`AS-JOB-001~func` 求职者 Agent 匹配职位并协商；`AS-005~func` 自主决定与候选 Agent 发起对话

**Step 6.2**: HR Agent 同时初筛多个候选人

- 操作: Bob HR Agent 收到多个 Agent 申请，同时进行初筛对话
- 预期: 初筛结论汇总到 HR 工作台
- 验证 AC: `AS-JOB-002-AC-2` 同时与多个候选人 Agent 初筛；`AS-JOB-002~func` 招聘方筛选简历并初筛；`AS-006~edge` 大量 Agent 同时联系

**Step 6.3**: 薪资协商 + 底线保护

- 操作: HR 报价低于 Alice 设定底线
- 预期: Alice Agent 拒绝并保留谈判空间，记录已披露信息；不会泄露真实底线数字
- 验证 AC: `AS-JOB-001-AC-3` 协商薪资保护主人底线；`JOB-004-AC-1` Agent 间薪资协商；`JOB-004~func` 四人群聊协商薪资

**Step 6.4**: 达成一致后引荐真人面试（四人群聊）

- 操作: 达成初步一致后引荐真人
- 预期: 四人群聊创建，Agent 自动转交后退居二线
- 验证 AC: `JOB-004-AC-2` 达成初步一致后引荐真人面试；`AS-009~func` 请求主人确认引荐

**Step 6.5**: 异常 —— 薪资底线被触碰 / 协商僵局 / 简历不足

- 操作: 模拟报价远低于底线；HR 简历信息不足
- 预期: Agent 礼貌终止；返回"信息不足"判断
- 验证 AC: `JOB-004~edge` 协商僵局、薪资超出底线；`AS-JOB-001~edge` 薪资底线被触碰；`AS-JOB-002~edge` 简历信息不足、初筛无法判断

### Phase 7: AgentAd 商家协商

**目的**: 在广告场景验证商家 Agent 与用户 Agent 协商优惠条件。

**Step 7.1**: 商家 Agent 寻找潜在客户并发起优惠协商

- 操作: 商家 Bob 的 Agent 主动联系 Alice Agent 推送优惠
- 预期: 协商房间创建；Agent 间往复议价
- 验证 AC: `AS-AD-002-AC-2` 与用户 Agent 协商优惠条件；`AD-004-AC-1` Agent 间协商优惠条件；`AS-AD-002~func` 商家寻找潜在客户推销优惠；`AD-004~func` Agent 与商家谈判争取优惠

**Step 7.2**: 达成双方满意方案

- 操作: 多轮报价后接受最终方案
- 预期: 生成优惠订单 / 写入需求-供给
- 验证 AC: `AD-004-AC-2` 达成双方满意的方案；`US-AGENT-002~func` 用户与 Agent 对话后成功创建需求和供给

**Step 7.3**: 异常 —— 谈判僵局 / 商家拒绝让步 / 无匹配用户

- 操作: 商家拒绝任何让步；用户 Agent 找不到目标用户
- 预期: 礼貌终止；不创建无效订单
- 验证 AC: `AD-004~edge` 谈判僵局、商家拒绝让步；`AS-AD-002~edge` 无匹配用户、协商失败

### Phase 8: 历史回看、举报与边界

**目的**: 验证按场景筛选历史、举报、Agent 创建边界。

**Step 8.1**: 按场景筛选 Agent 对话历史

- 操作: 进入"我的对话"，按交友/求职/广告筛选
- 预期: 列表显示双方 Agent 头像、匹配度评分、对话摘要
- 验证 AC: `US-AGENT-005-AC-1` 按场景筛选对话记录；`US-AGENT-005-AC-2` 显示对话双方 Agent 和匹配度评分；`US-AGENT-005~func` 按场景筛选并查看历史

**Step 8.2**: 在对话界面举报不当行为

- 操作: 点击对话右上角"举报"，提交举报
- 预期: 举报记录入库，相关 Agent 被打上风险标记
- 验证 AC: `US-AGENT-016-AC-1` 可在对话界面举报；`US-AGENT-016~func` 用户在对话界面举报不当行为

**Step 8.3**: Agent 创建边界 —— 空输入 / 无意义输入 / 超长对话；历史加载异常

- 操作: 训练 Agent 时输入空字符串、随机字符；对话拉到 1000+ 条；离线状态下进入历史
- 预期: 提示输入有效内容；分页加载稳定；异常时显示"加载失败 重试"
- 验证 AC: `US-AGENT-002~edge` 空输入、无意义输入、超长对话；`US-AGENT-005~edge` 历史记录缺失、加载异常

## 环境启动

```bash
# 1. 启动数据库与缓存
cd /Users/z/projects/bridgeai
docker compose up -d postgres redis

# 2. 启动后端 API（带 WebSocket）
cd backend && uv run uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

# 3. 启动前端
cd frontend && pnpm install && pnpm dev

# 4. 准备测试账号
cd backend && uv run python scripts/seed_demo_users.py --users alice,bob

# 5. 打开两个浏览器（或一个普通 + 一个隐身）
# Alice: http://localhost:3000  （登录 alice@example.com）
# Bob:   http://localhost:3000  （登录 bob@example.com）

# 6. 启用调试面板
# DevTools → Application → LocalStorage → 设置 BRIDGEAI_DEBUG=1
# 用于查看 whisper 通道、披露日志、信任分实时计算

# 7. 故障注入（可选）
# 切断 AI 建议服务用于 3.3 / 8.3：
docker compose stop ai-suggester
# 网络丢包用于 1.3 / 5.4：
# Chrome DevTools → Network → Throttling → Offline
```
