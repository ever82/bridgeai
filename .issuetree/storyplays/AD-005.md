# Story Playbook: 一键购买和商家评分

**Story**: `AD-005` | **Narrative**: 一键购买，购买后评价商家。

## 验收旅程概览

| Phase | 焦点           | 关键验证                       |
| ----- | -------------- | ------------------------------ |
| 1     | 积分系统数据层 | 数据库表、账户模型、积分余额   |
| 2     | 积分交易服务   | 积分获取/消耗/冻结/转账原子性  |
| 3     | 积分 API 接口  | 余额查询、交易记录、统计接口   |
| 4     | 积分钱包 UI    | 钱包首页、交易时间轴、收支概览 |
| 5     | 一键购买流程   | 商家跳转、优惠应用、购买确认   |
| 6     | 商家评分       | 评价提交、评分展示             |
| 7     | 实时通信       | Socket.io 连接、房间、广播     |

## 前置条件

- [ ] 数据库已迁移: `pnpm db:migrate`
- [ ] 测试数据已填充: `pnpm db:seed`
- [ ] 开发服务器运行中: `pnpm dev`
- [ ] 积分账户预置: 测试用户 A 有 500 积分，测试用户 B 有 1000 积分
- [ ] 商家 Agent 已配置并在线
- [ ] Redis 服务运行（支持多节点 Socket.io）
- [ ] 移动端 App 已安装或 Web 端可访问

## 旅程步骤

---

### Phase 1: 积分系统数据层

**目的**: 验证积分账户数据库模型和表结构正确性

**Step 1.1**: 验证数据库表结构

- 操作: 连接数据库，检查 PointsAccount、PointsTransaction、PointsFreeze 表
- 预期:
  - PointsAccount 表存在 user_id、balance、total_earned、total_spent、frozen_amount 字段
  - PointsTransaction 表存在 transaction_id、type、amount、description、scene_type 字段
  - PointsFreeze 表存在 freeze_id、amount、status、expire_at 字段
  - 账户有唯一性约束，余额有非负检查
- 验证 AC: `ISSUE-CR003a~c1` 数据库设计完整性

**Step 1.2**: 验证账户创建和关联

- 操作: 查询测试用户 A 的积分账户
- 预期: 账户存在，user_id 关联正确，初始余额为 500
- 验证 AC: `ISSUE-CR003a~c1` 账户唯一性约束和用户关联

---

### Phase 2: 积分交易核心服务

**目的**: 验证积分操作的原子性、安全性和规则配置

**Step 2.1**: 验证积分获取（签到奖励）

- 操作: 调用签到接口，奖励 10 积分
- 预期: 余额从 500 增加到 510，产生一条类型为 EARN 的交易记录
- 验证 AC: `ISSUE-CR003b~c1` 积分获取接口

**Step 2.2**: 验证积分消耗（查看照片）

- 操作: 用户 A 用 50 积分查看 VisionShare 照片
- 预期: 余额从 510 减少到 460，交易记录显示 SPEND 类型
- 验证 AC: `ISSUE-CR003b~c1` 积分消耗接口

**Step 2.3**: 验证积分冻结（交易担保）

- 操作: 发起购买请求，冻结 100 积分
- 预期: 账户 frozen_amount 增加 100，实际可用余额相应减少
- 验证 AC: `ISSUE-CR003b~c1` 积分冻结/解冻

**Step 2.4**: 验证并发安全（防止超卖）

- 操作: 模拟两个并发请求同时消耗同一账户余额
- 预期: 只成功执行一次，余额不会变成负数，事务回滚正确
- 验证 AC: `ISSUE-CR003b~c1` 事务安全性

**Step 2.5**: 验证积分规则配置

- 操作: 查询每日获取上限配置
- 预期: 规则可查询，签到奖励受每日上限限制
- 验证 AC: `ISSUE-CR003b~c2` 积分获取/消耗规则配置
