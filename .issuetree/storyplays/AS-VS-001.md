# Story Playbook: VisionShare Agent 配置与互动

**Story**: `AS-VS-001`
**Narrative**: 主人想看某地实时情况，我（Agent）提炼需求并寻找响应者。

## 验收旅程概览

作为 VisionShare 用户，与 Agent 协作完成一次完整的视觉需求发布：从自然语言描述"我想看某地某时某景"，经历 AI 多适配器解析意图、熔断保护下路由降级、Agent 设置地域与信用过滤条件、地图查看附近候选响应者、筛选并接受任务的端到端旅程。

| 阶段                | 核心操作                                | 验证重点                    |
| ------------------- | --------------------------------------- | --------------------------- |
| Phase 1: 需求提出   | 主人对 Agent 说"我想看深圳湾大桥的日落" | 多 LLM 适配器解析、意图提取 |
| Phase 2: 智能配置   | Agent 配置过滤条件并搜索响应者          | 路由策略、降级、熔断器      |
| Phase 3: 响应者发现 | 主人浏览地图上的候选响应者              | 地图、筛选、推荐            |
| Phase 4: 接单体验   | 主人选择响应者、进入拍照                | 接单流程、状态管理          |
| Phase 5: 异常处理   | 地点无法解析 / 无附近响应者             | 边缘场景容错                |

**总 AC 数**: 22 | **目标覆盖**: 22 (100% 有机覆盖)

### Phase 1: 需求提出 — Agent 解析主人意图

**目的**: 用户以自然语言对 Agent 描述需求，AI 多适配器协同完成意图解析和地点提取，熔断器全程保护。

---

**Step 1.1**: 唤醒 VisionShare Agent 并发送自然语言请求

- 操作: 在 VisionShare 场景中，点击 Agent 对话入口，发送消息："我想看深圳湾大桥下午四点的日落，要能看到海面和对面高楼的剪影"
- 预期:
  - 消息气泡正常显示
  - Agent 回复正在解析，显示流式打字效果或加载动画
  - 解析成功后，Agent 展示结构化理解结果
- 截图验证: Agent 对话界面（解析结果卡片）
- 验证 AC:
  - `AS-VS-001-AC-1` 解析地点意图
  - `VS-001-AC-1` 用户输入自然语言描述（地点+意图）
  - `ISSUE-AI001~c1` OpenAI/Claude/文心一言统一适配器接口
  - `ISSUE-AI001~c5` LLM API 密钥安全存储与流式响应

---

**Step 1.2**: 验证 AI 适配器路由和熔断保护

- 操作: 在开发者工具或日志中，查看本次请求的适配器选择和熔断状态
- 预期:
  - 请求经过统一适配器接口分发
  - 路由策略按配置选择合适的模型（成本优先/质量优先）
  - 熔断器状态为"关闭"（请求正常）
  - 熔断事件日志中有记录（如有失败）
- 截图验证: 熔断器状态面板 / 请求日志
- 验证 AC:
  - `ISSUE-AI001~c1` 适配器实现（统一接口抽象层、模型能力注册表）
  - `ISSUE-AI001~c2` 熔断器实现（失败率监控、状态管理、自动恢复探测）
  - `ISSUE-AI001~c3` 路由策略（成本路由、延迟路由、质量路由、权重负载均衡）
  - `ISSUE-AI001~c6` 监控系统（请求延迟、成功率/失败率、Token 使用量、熔断日志）

**Step 1.3**: 验证降级策略（模拟模型不可用）

- 预期:
  - 系统检测到主适配器不可用，自动切换到备用适配器

  - 响应中包含降级标记或日志
- 截图验证: 降级日志或响应头标记
- 验证 AC:
  - `ISSUE-AI001~c4` 降级策略（模型降级、提供商降级、功能降级、缓存回退、异步队列兜底）
  - `ISSUE-AI001~c2` 熔断器状态自动转换（关闭 → 打开 → 半开）

---

**Step 1.4**: 查看 Agent 提取的需求结构

- 操作: 在 Agent 回复中，展开"需求详情"卡片
- 预期:
  - L1 层: 地点（深圳湾大桥）、时间（16:00）、类型（日落风景）

**Step 2.2**: 微调地域过滤

### Phase 4: 接单体验 — 接受任务并进入拍照

  - 检查不通过：显示原因（如「积分不足，请充值」或「信用分不满足」）
- 截图验证: 拍照界面或资格不满足提示

**Step 4.3**: 接单后状态同步

- 预期:

### Phase 5: 异常处理 — 边缘场景验证

**目的**: 验证系统在无法解析地点、无附近响应者等异常情况下的容错和用户体验。

---

**Step 5.1**: 无法解析地点异常

- 操作: 向 Agent 发送模糊请求："我想看那个地方，就是之前去过的那个"（无具体地点）
- 预期:
  - Agent 询问澄清：「未能识别具体地点，请提供更详细的位置描述」
  - 不崩溃，不过度消耗 Token
  - 记录解析失败日志
- 截图验证: Agent 澄清回复
- 验证 AC:
  - `AS-VS-001~edge` 异常：无法解析地点
  - `VS-001~edge` 异常：无法解析地点
  - `ISSUE-AI001~c6` 监控系统（熔断事件日志、请求延迟）

---

**Step 5.2**: 附近无候选响应者异常

- 操作: 回到过滤条件配置，设置极小范围（100m）+ 极高信用分（98），点击确认搜索
- 预期:
  - 系统显示：「附近暂无符合条件的响应者，建议扩大搜索范围」
  - 提供「一键扩大范围」按钮
  - 允许发布（只是可能长时间无响应）
- 截图验证: 无匹配结果提示
- 验证 AC:
  - `AS-VS-001~edge` 异常：无候选响应者
  - `VS-001~edge` 异常：附近无匹配用户
  - `ISSUE-VS002~c5` 推荐系统（替代推荐）

---

**Step 5.3**: 位置异常和信用过滤绕过

- 操作:
  a. 模拟 GPS 异常（关闭定位服务），尝试查看附近任务
  b. 模拟修改请求中的信用分参数（绕过前端验证），尝试接高信用分任务
- 预期:
  a. GPS 异常时显示：「无法获取位置，请开启定位服务」，地图不可用
  b. 后端应拒绝信用分不符的接单请求（即使前端参数被篡改）
  - 后端返回 403 或相应错误码
- 截图验证: GPS 提示 / 后端拒绝日志
- 验证 AC:
  - `VS-002~edge` 异常：位置异常、信用过滤绕过
  - `ISSUE-VS002~c4` 接单流程（接单资格检查必须在后端验证）

---

**Step 5.4**: LLM 服务完全不可用（熔断器全开）

- 操作: 在开发者工具中禁用所有 LLM 适配器，然后尝试向 Agent 发送新的视觉需求
- 预期:
  - 熔断器状态变为"打开"
  - Agent 返回降级回复：「当前服务繁忙，请稍后再试」
  - 异步队列可能接管（如果实现了异步兜底）
  - 监控系统记录熔断事件
- 截图验证: 熔断器全开状态和降级回复
- 验证 AC:
  - `ISSUE-AI001~c2` 熔断器（熔断状态管理、多维度熔断策略、自动恢复探测）
  - `ISSUE-AI001~c4` 降级策略（异步队列兜底、缓存响应回退）
