# Story Playbook: 面试后评分

**Story**: `JOB-005` | **Narrative**: 面试完成后双方评分。

## 验收旅程概览

| Phase                 | 核心验证                         | AC 覆盖数 |
| --------------------- | -------------------------------- | --------- |
| Phase 1: 面试流程收尾 | 面试结束后评分弹窗触发           | 1         |
| Phase 2: 评分操作     | 多维度评分 + 文字评价提交        | 3         |
| Phase 3: 信用分联动   | 评分影响信用分计算               | 2         |
| Phase 4: 评价管理     | 查看/回复/统计评价               | 2         |
| Phase 5: 异常处理     | 恶意评分/重复评分/未面试评分拦截 | 3         |

**总计 AC 覆盖: 11 / 57 (19.3%)**

## 前置条件

- 两台已登录移动设备（候选人 Alice / 招聘方 Bob）
- 至少完成一次 AgentJob 场景面试会话
- 候选人 Alice 和招聘方 Bob 的信用分初始值已记录
- 后端 API 服务运行中（POST /api/v1/reviews 等接口就绪）
- 通知服务（Push / In-App）可用

## 旅程步骤

---

### Phase 1: 面试结束触发评分

**目的**: 验证面试完成后自动弹出评分界面（交流结束后弹出评分界面）

**Step 1.1**: 模拟面试会话结束

- 操作: 在候选人 Alice 设备上点击"结束面试"按钮，会话状态流转为 completed
- 预期: 面试结束确认弹窗出现，显示"感谢参与面试，请对本次面试评分"
- 验证 AC: `JOB-005~func` 面试完成后双方评分

**Step 1.2**: 评分弹窗自动弹出

- 操作: 等待会话结束动画完成（约 2 秒）
- 预期: 评分弹窗从底部滑入，标题为"为面试评分"，显示被评价方（招聘方 Bob）信息
- 截图验证: 弹窗包含星级评分组件（多维度：诚信度、礼貌度、响应度、满意度）和文字输入框
- 验证 AC: `DATE-005-AC-1` 交流结束后弹出评分界面

---

### Phase 2: 提交多维度评分

**目的**: 验证多维度评分能力及评价提交流程

**Step 2.1**: 填写多维度星级评分

- 操作: 候选人 Alice 在弹窗中依次点击各维度 5 星：
  - 诚信度: 点击第 5 颗星
  - 礼貌度: 点击第 5 颗星
  - 响应度: 点击第 4 颗星
  - 满意度: 点击第 5 颗星
- 预期: 每个维度星级高亮，选中星变为实心金色，未选中为灰色轮廓
- 截图验证: 4 个评分维度均显示选中状态
- 验证 AC: `US-AGENT-010-AC-1` 多维度评分（诚信度、礼貌度、响应度、满意度）

**Step 2.2**: 填写文字评价

- 操作: 在文字输入框中输入："面试官非常专业，流程清晰，对岗位要求说明详细。"
- 预期: 文字输入框显示已输入内容，字符计数正常（支持 500 字以内）
- 验证 AC: `ISSUE-CR002a~c2` 评分提交接口（评价内容需通过权限验证）

**Step 2.3**: 提交评价

- 操作: 点击"提交评价"按钮
- 预期:
  - 按钮变为加载状态（禁用 + spinner）
  - 成功后弹窗关闭，页面显示 toast: "评价已提交，感谢您的反馈"
  - 跳转至面试结果页或会话列表页
- 截图验证: toast 提示 + 页面状态更新
- 验证 AC: `DATE-005~func` 约会完成后双方评分，影响信用分

**Step 2.4**: 招聘方侧评分

- 操作: 切换至招聘方 Bob 设备，等待其会话状态更新
- 预期: Bob 设备同样收到评分弹窗（独立触发），可对候选人 Alice 评分
- 截图验证: Bob 设备评分弹窗正常显示候选人信息
- 验证 AC: `JOB-005~func` 面试完成后双方评分（双向验证）

---

### Phase 3: 信用分联动验证

**目的**: 验证评分后信用分实时更新（双向评分影响信用分）

**Step 3.1**: 查看候选人信用分变化

- 操作: 候选人 Alice 点击"我的 → 信用分"入口
- 预期:
  - 信用分数值更新（若 Bob 也提交了评分，两个方向的评分均影响信用分）
  - 信用分变化记录显示 "+N 分（来自招聘方好评）" 标签
  - 或 "-N 分（来自招聘方差评）" 视评分结果而定
- 截图验证: 信用分详情页显示评分来源和变化值
- 验证 AC: `JOB-005-AC-1` 双向评分影响信用分、`US-AGENT-010-AC-2` 评分影响信用分

**Step 3.2**: 查看招聘方信用分变化

- 操作: 招聘方 Bob 点击"我的 → 信用分"入口
- 预期:
  - 信用分变化记录显示评分方为"候选人 Alice"
  - 包含时间戳和对应的评分维度摘要
- 验证 AC: `ISSUE-CR002c~c1` 评价对信用分的影响（收到好评增加信用分 / 收到差评扣除信用分）

**Step 3.3**: 验证通知推送

- 操作: 查看 Bob 设备通知栏（或应用内通知列表）
- 预期:
  - 收到新评价推送通知："您收到一条来自 Alice 的新评价"
  - 待评价提醒（若 Bob 尚未提交评分）："您还有一条待评价，快去评分吧"
- 截图验证: 通知列表显示对应通知项
- 验证 AC: `ISSUE-CR002c~c2` 评价相关通知（收到新评价推送通知 + 待评价提醒）

---

### Phase 4: 评价管理与查看

**目的**: 验证评价列表、详情、回复功能

**Step 4.1**: 查看收到的评价列表

- 操作: 招聘方 Bob 进入"我的 → 收到的评价"
- 预期:
  - 列表页显示来自 Alice 的评价卡片
  - 卡片包含：评分维度星级展示、文字评价内容、提交时间
  - 按时间倒序排列
- 截图验证: 评价卡片组件样式完整
- 验证 AC: `ISSUE-CR002b~c1` 移动端评价UI（评价列表页面 + 评价卡片组件）

**Step 4.2**: 查看评价详情页

- 操作: Bob 点击评价卡片进入详情页
- 预期:
  - 展示完整评价信息（多维度评分雷达图或条形图）
  - 显示评价者信息（Alice）
  - 底部有"回复评价"按钮（如有回复则显示）
- 截图验证: 评价详情页含评分分布可视化
- 验证 AC: `ISSUE-CR002b~c1` 移动端评价UI（评价详情页含回复）

**Step 4.3**: 回复评价

- 操作: Bob 在评价详情页点击"回复评价"，输入："感谢您的认可，祝求职顺利。"
- 预期:
  - 回复内容提交成功，显示在原评价下方
  - 回复时间戳正常
- 截图验证: 回复内容展示在评价下方，带"招聘方回复"标签
- 验证 AC: `ISSUE-CR002a~c2` PUT /api/v1/reviews/:id/reply 回复评价接口

**Step 4.4**: 查看发出的评价列表

- 操作: Alice 进入"我的 → 发出的评价"
- 预期:
  - 列表显示 Alice 已提交的评价
  - 显示评价对象（Bob）及状态（已回复/待回复）
- 验证 AC: `ISSUE-CR002a~c2` GET /api/v1/reviews/given 获取发出的评价

**Step 4.5**: 查看评价统计

- 操作: Alice 进入"我的 → 评价统计"
- 预期:
  - 平均分展示（各维度平均分）
  - 评价数量统计
  - 评价分布图（如饼图或条形图）
- 截图验证: 评价统计页包含平均分 + 分布图
- 验证 AC: `ISSUE-CR002b~c1` 移动端评价UI（评价统计展示：平均分、分布图）

---

### Phase 5: 异常处理与安全验证

**目的**: 验证恶意评分拦截、重复评分拦截、未面试评分拦截

**Step 5.1**: 未完成面试尝试评分（应被拦截）

- 操作: 招聘方 Bob 尝试对未完成面试的候选人发起评分（直接调用 POST /api/v1/reviews）
- 预期:
  - API 返回 403 Forbidden 或 400 Bad Request
  - 错误信息："当前状态不允许评价，需完成面试后操作"
- 截图验证: API 错误响应或前端拦截提示
- 验证 AC: `JOB-005~edge` 异常：未面试就评分、`ISSUE-CR002a~c2` 评价权限验证（交易完成后才可评价）

**Step 5.2**: 重复评分拦截

- 操作: 候选人 Alice 已提交对 Bob 的评分，再次尝试提交（同一 session）
- 预期:
  - API 返回 409 Conflict 或 400
  - 错误信息："您已评价过该用户，请勿重复评价"
- 验证 AC: `DATE-005~edge` 异常：恶意评分、重复评分、`US-AGENT-010~edge` 异常：恶意评分、重复评分

**Step 5.3**: 敏感词内容审核

- 操作: 候选人 Alice 重新提交一条含敏感词的评价："该职位是骗人的，大家不要上当！！！"
- 预期:
  - 提交时前端给出警告（若前端有敏感词过滤）
  - 或后端 AI 审核拒绝，返回："评价内容未通过审核，请修改后重试"
  - 评价被标记为待审核状态，不对外展示
- 截图验证: 审核拒绝提示或评价状态为"审核中"
- 验证 AC: `ISSUE-CR002a~c3` 评价内容审核机制（敏感词过滤和AI内容审核）

**Step 5.4**: 恶意刷评价检测

- 操作: 通过脚本或快速连续提交多个评价（模拟同一用户多次评价不同用户）
- 预期:
  - 系统检测到异常行为
  - 部分评价被标记为可疑
  - 或触发反作弊机制，限制评价频率
- 验证 AC: `ISSUE-CR002a~c3` 评价内容审核机制（恶意刷评价检测）

**Step 5.5**: 评价删除（管理员）

- 操作: 管理员调用 DELETE /api/v1/reviews/:id 删除违规评价
- 预期:
  - 删除成功返回 204
  - 该评价从列表中消失
- 验证 AC: `ISSUE-CR002a~c2` DELETE /api/v1/reviews/:id 删除评价（管理员）
