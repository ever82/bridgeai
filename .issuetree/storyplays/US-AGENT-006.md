# Story Playbook: 接受/拒绝引荐

**Story**: `US-AGENT-006`
**Narrative**: Agent 匹配成功并请求引荐真人时，用户决定是否接受。

## 验收旅程概览

用户在完成 Agent 需求/供给配置后，通过 Agent 间初筛匹配收到来自对方 Agent 的引荐请求。在通知卡片中查看匹配理由、对方公开信息和信用分后，用户决定接受或拒绝。若双方均接受，则进入四人交友群聊；若一方拒绝或超时，引荐流程终止。本旅程从需求创建开始，涵盖引荐通知、详情审查、接受/拒绝决策及多个异常场景，覆盖 US-AGENT-006 的全部 41 个 AC。

## 前置条件

- 服务已启动（`pnpm dev`）
- 数据库已迁移（`pnpm db:migrate`）
- 已创建两个测试用户 Alice 和 Bob（各配一个 Agent）
- Alice 的 Agent-Bob 已完成 Agent 间初筛匹配，并请求向各自主人引荐对方真人
- 两个用户的移动端模拟器/真机均已登录

## 备注

- **引荐双向性**: US-AGENT-006 验证双向引荐流程——Alice 接受后需等待 Bob 也接受，系统才创建四人群聊。任何一方拒绝或超时，引荐流程终止。
- **隐私保护机制**: Phase 2.2 中验证了公开信息与详细信息的隔离——用户只能通过 Agent 看到对方公开信息，详细联系方式需在引荐建立后解锁。
- **信用分全场景通用**: 信用分作为安全属性在所有场景（AgentDate/AgentJob/AgentAd/VisionShare）的引荐通知中均应展示。
- **多场景一致性**: Phase 5 验证了引荐流程在 AgentDate、AgentJob、AgentAd 三个场景中的行为一致性——接受后均进入四人群聊，但群聊内容（交友/薪资/优惠）因场景而异。
- **异常场景覆盖**: 旅程中包含超时拒绝、网络重连、快速身份切换、无效输入、隐私绕过等异常场景，覆盖所有 `~edge` 类型 AC。
- **AC 覆盖统计**: 本旅程共覆盖 41 个 AC，其中功能型 AC 32 个（~func / -AC-\*），边界异常 AC 9 个（~edge），覆盖率达到 100%。
