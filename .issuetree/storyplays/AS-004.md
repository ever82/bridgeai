# Story Playbook: 我收到实时匹配任务

**Story**: `AS-004` | **Narrative**: 主人表达需求/供给，系统返回候选，我批量评估筛选。

## 验收旅程概览

| Phase               | 核心流程                              | 覆盖 AC 数 |
| ------------------- | ------------------------------------- | ---------- |
| 1. 需求表达与触发   | 主人输入需求 -> Agent解析 -> 激活匹配 | 8          |
| 2. 多维过滤筛选     | 地域/属性/信用 -> 批量读取候选        | 28         |
| 3. 匹配度计算与排序 | 计算分数 -> 排序推荐 -> Top N         | 12         |
| 4. 评估决策与引荐   | 展示推荐 -> 发起对话 -> 请求引荐      | 9          |
| 5. 异常场景处理     | 无候选/过严过滤/服务异常              | 7          |

**总 AC 覆盖**: 64 / 164 (39%) - 核心旅程全覆盖，边缘功能按需验证

## 前置条件

1. 测试账号: `test_user_001` (主人角色)
2. 测试 Agent 已初始化，具备基础画像
3. 测试环境中存在多个候选 Agent (至少 20 个)
4. 候选 Agent 分布在不同地域、有不同属性和信用分
5. 推送/订阅服务已启动 (WebSocket 可用)

## 旅程步骤

### Phase 1: 需求表达与实时触发

**目的**: 验证主人通过自然语言表达需求后，系统正确解析并触发实时匹配

---

**Step 1.1**: 主人通过 Agent 表达需求

- 操作: 用户在 Agent 对话界面输入自然语言需求
  ```
  "我想找一个在浦东的摄影师，能做活动摄影，预算 500-800/小时"
  ```
- 预期:
  - Agent 理解并解析需求关键词 (地点、技能、预算范围)
  - 显示需求确认卡片，包含解析结果
  - 提供"确认发布"按钮
- 验证 AC:
  - `AS-002~func` Agent 解析主人自然语言输入
  - `AS-002-AC-3` 确认后存储并触发实时匹配
  - `US-AGENT-013-AC-1` Agent 展示对方公开的需求和供给概要

**Step 1.2**: 主人确认需求，触发匹配

- 操作: 点击"确认发布"按钮
- 预期:
  - 显示加载状态 (正在为您寻找匹配...)
  - 需求被存储到数据库
  - 触发实时匹配查询
- 验证 AC:
  - `AS-003~func` Agent 理解主人供给能力，记录为供给档案
  - `AS-003-AC-3` 激活供给状态等待匹配查询
  - `ISSUE-M001~c4` WebSocket 推送实时更新

**Step 1.3**: 实时接收匹配结果推送

- 操作: 等待 WebSocket 推送，观察通知
- 预期:
  - 收到实时推送通知 (有 15 位候选人符合条件)
  - 显示候选人卡片列表
  - 列表按相关性排序
- 验证 AC:
  - `VS-001~func` 系统解析并推送给附近符合条件的用户
  - `ISSUE-M001~c4` 实时更新机制 (WebSocket推送)
  - `ISSUE-M001~c4` 查询结果缓存

### Phase 2: 多维过滤筛选

**目的**: 验证地域、属性、信用三重过滤系统正确工作

---

**Step 2.1**: 验证地域过滤

- 操作: 在候选人列表顶部查看过滤条件栏，点击"地点"筛选

- 验证 AC:
  - `ISSUE-C003~c1` 地理数据 (省市区编码、经纬度)
  - `ISSUE-C003~c3` 距离筛选 (基于经纬度的距离计算)
  - `ISSUE-C003~c6` Agent 位置筛选 API
  - `AS-VS-001-AC-3` 筛选信用高、距离近的候选
  - `VS-002-AC-1` 只显示符合地域过滤条件的任务

**Step 2.2**: 验证属性过滤

- 操作: 点击"属性"筛选，添加条件:
  - 技能包含: "摄影"
  - 评分 >= 4.5
  - 接单数 >= 10
- 预期:
  - 候选人数量从 8 降至 4
  - 每个人卡片显示技能标签和评分
- 验证 AC:
  - `ISSUE-C004~c1` 查询 DSL (条件表达式、操作符 in/contains/gt)
  - `ISSUE-C004~c2` DSL 到 SQL 转换
  - `ISSUE-C004~c4` 按属性筛选 Agent 接口
  - `AS-004-AC-2` 批量读取候选公开信息
  - `AS-005-AC-1` 根据过滤规则筛选候选

**Step 2.3**: 验证信用过滤

- 操作: 点击"信用"筛选，设置最低门槛: 600 分
- 预期:
  - 候选人数量从 4 降至 3
  - 每个人卡片显示信用分徽章

- 验证 AC:

**Step 2.4**: 查看候选人详细信息

- 操作: 点击候选人卡片展开详情
- 预期:
  - 显示完整公开档案 (姓名、头像、简介、技能)
  - 显示近期作品/案例
  - 显示历史评价摘要
- 验证 AC:
  - `US-AGENT-013~func` 通过 Agent 浏览对方公开需求和供给
  - `US-AGENT-013-AC-1` Agent 展示对方公开的需求和供给概要

### Phase 4: 评估决策与引荐

**目的**: 验证 Agent 自主评估后向主人请求引荐确认

---

**Step 4.1**: Agent 评估并发送建议

- 操作: 查看 Agent 建议卡片
- 预期:
  - 显示 Top 3 推荐及理由
  - 显示匹配度评估说明
  - 显示"建议联系"或"观望"标签
- 验证 AC:
  - `AS-DATE-001-AC-2` 批量评估匹配度
  - `AS-DATE-002-AC-3` 双方合适后请求引荐
  - `DATE-003-AC-2` 匹配度达标后请求引荐
  - `JOB-004~func` 双方 Agent 和用户在四人群聊中协商薪资

**Step 4.2**: Agent 发起对话请求

- 操作: 点击"让 Agent 联系"按钮
- 预期:
  - Agent 以自己名义向候选 Agent 发送消息
  - 消息内容体现主人需求
  - 等待候选 Agent 回复
- 验证 AC:
  - `AS-005-AC-2` 自主发起对话请求
  - `AS-005~func` Agent 自主决定与符合条件的候选 Agent 发起对话
  - `US-AGENT-007-AC-2` 点击"让 Agent 聊"按钮后 Agent 以自己名义发言

**Step 4.3**: 同时与多个候选对话

- 操作: 对多个候选重复 Step 4.2
- 预期:
  - 可以同时联系 3 个候选
  - 每个对话独立进行
  - 显示对话列表标签页
- 验证 AC:
  - `AS-005-AC-3` 同时与多个候选保持独立对话
  - `US-AGENT-009-AC-1` 私聊建议显示在侧边栏或独立区域

**Step 4.4**: 查看 Agent 私聊建议

- 操作: 在对话侧边栏查看建议区域
- 预期:
  - 显示话题推荐
  - 显示风险提示
  - 显示对方意图分析
- 验证 AC:
  - `AS-008-AC-1` 实时分析对方意图和态度
  - `AS-008-AC-2` 发送策略建议、风险提示、话题推荐
  - `US-AGENT-009-AC-2` 建议包括话题推荐、风险提示、对方意图分析

**Step 4.5**: 匹配度达标，请求引荐确认

### Phase 5: 异常场景处理

**目的**: 验证系统在边界和异常情况下的表现

---

**Step 5.1**: 无候选用户场景

- 操作: 设置极端过滤条件 (如: 上海市 + 信用分 >= 900 + 技能 = "量子计算")
- 预期:
  - 显示空状态页面
  - 提示: "未找到符合条件的候选人"
  - 提供"放宽条件"建议按钮
- 验证 AC:
  - `AS-004~edge` 异常：无候选、匹配算法出错
  - `DATE-002~edge` 异常：无匹配用户、过滤条件过严
  - `AS-DATE-001~edge` 异常：过滤后无候选
  - `VS-001~edge` 异常：无法解析地点、附近无匹配用户

**Step 5.2**: 过滤条件过严警告

- 操作: 设置多个严格过滤条件后查看结果
- 预期:
  - 候选人数量明显减少时显示警告
  - 提示: "条件过严，可能错过优质候选"
- 验证 AC:
  - `JOB-003~edge` 异常：单向匹配、过滤条件变更
  - `ISSUE-C004~c3` 过滤 UI 提供条件建议

**Step 5.3**: 大量 Agent 同时联系

- 操作: 被多个候选 Agent 同时发起对话
- 预期:
  - 系统限流处理，不超过 5 个同时请求
  - 显示通知队列
- 验证 AC:
  - `AS-006~edge` 异常：大量 Agent 同时联系
  - `ISSUE-M001~c3` 资源管理和限流

**Step 5.4**: 尝试查看非公开信息

- 操作: 点击"查看更多资料"按钮
- 预期:
  - 显示隐私保护提示
  - 提示: "该信息仅对已引荐用户可见"
- 验证 AC:
  - `US-AGENT-013~edge` 异常：尝试查看详细信息

**Step 5.5**: AI 服务降级处理

- 操作: 模拟 AI 服务不可用场景
- 预期:
  - 显示基础匹配结果 (无详细分析)

  - `AS-008~edge` 异常：建议为空、AI 服务降级
  - `ISSUE-M001~c3` 执行计划优化 (降级策略)

## 环境启动

### 1. 启动后端服务

```bash
cd /Users/z/projects/bridgeai

# 启动 PostgreSQL (PostGIS)
docker-compose up -d postgres

# 启动后端 API
pnpm --filter @bridgeai/api dev

# 启动 WebSocket 服务
pnpm --filter @bridgeai/ws dev
```

### 2. 启动前端

```bash
# 移动端 App
pnpm --filter @bridgeai/mobile dev

# 或 Web 端
pnpm --filter @bridgeai/web dev
```

### 3. 初始化测试数据

```bash
# 运行数据库迁移
pnpm --filter @bridgeai/api db:migrate

# 播种测试数据
pnpm --filter @bridgeai/api db:seed:test
```

### 4. 启动测试环境

```bash
# E2E 测试
pnpm --filter @bridgeai/e2e test

# 或手动验证
open http://localhost:3000
```
