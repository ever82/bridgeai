# Story Playbook: 创建主人的供给

**Story**: `AS-003` | **Narrative**: 主人告诉我他能提供什么，我记录为供给档案。

## 验收旅程概览

本剧本以"小明（求职者 + 摄影爱好者）"和"小红（HR + 商家）"两位主人为视角，
覆盖从 Agent 初始化、用自然语言表达供给能力、Agent 自动提炼三层信息模型、
供给档案激活、实时匹配被发现、Agent 间聊天协作、向主人请求引荐、
四人群聊切换模式、退出与举报的完整供给生命周期。

旅程顺序遵循**真实使用路径**：注册 → 配置（L1/L2/L3 供给侧）→ 自然语言表达供给
→ AI 提炼确认 → 存储/激活 → 被候选方匹配查询发现 → Agent 间协商 → 建议
→ 披露策略 → 引荐 → 群聊 → 退出/举报，并在每个阶段混入 4 个场景
（VisionShare/AgentDate/AgentJob/AgentAd）的差异化覆盖与异常路径。

- 总 Phase: 9
- 总 Step: 34
- 唯一 AC 数: 121
- 覆盖场景: 通用 + VS + Date + Job + Ad

### Phase 2: 自然语言创建供给档案（AS-003 主线）

**目的**: 验证主人用自然语言表达供给能力、Agent 自动提炼并记录为供给档案的完整链路。

**Step 2.2**: Agent 确认并激活供给档案

- 操作: 点击"确认激活供给"按钮。
- 预期: 卡片折叠为 timeline 条目；后端写入 supply 表 (scene=job, status=active)；
  触发 WebSocket 实时通知匹配系统该供给已上线；同步触发对偶的需求侧匹配

- 验证 AC: `AS-003-AC-3` 激活供给状态等待匹配查询；`US-AGENT-002-AC-5` 自动保存到对应场景；

  `AS-002~func` 需求侧自然语言提炼；`AS-002-AC-3` 需求确认后存储+触发实时匹配。

- 验证 AC: `AS-003~edge` 供给为空、激活失败。

## 环境启动

```bash
# 1) 安装依赖
cd /Users/z/projects/bridgeai
pnpm install

# 2) 启动数据库（需要本地 Postgres + PostGIS 已就绪）
pnpm db:migrate
pnpm db:seed

# 3) 启动后端 + 移动端（Expo Web）
pnpm dev    # turbo 并行启动 apps/server 与 apps/mobile

# 4) 单独运行端到端验收（可选）
pnpm test:e2e:web      # web 端到端
pnpm test:e2e          # mobile 端到端

# 5) 构建生产
pnpm build
```

> `e2e_screenshots/AS-003/Phase{n}-Step{m}.png`，便于人工或 demo 复核。
