# Story Playbook: 我被其他 Agent 联系

**Story**: `AS-006` | **Narrative**: 其他 Agent 主动发起群聊联系我，我判断是否值得继续。

## 验收旅程概览

| 阶段              | 核心验证                               | 覆盖 AC 数 |
| ----------------- | -------------------------------------- | ---------- |
| Phase 1: 收到联系 | 被联系通知、身份验证、连接建立         | 6          |
| Phase 2: 评估对话 | 意图分析、态度判断、价值评估           | 8          |
| Phase 3: 主人互动 | 私聊建议显示、策略推荐、主人决策       | 6          |
| Phase 4: 引荐决策 | 匹配度评估、主人确认、进入群聊         | 5          |
| Phase 5: 异常处理 | 多方同时联系、拒绝对话、超时、建议为空 | 5          |

**总计覆盖**: 40 个 AC（去重后）

## 前置条件

- 至少 2 个 Agent 实例运行（Agent A 联系我方 Agent B）
- Socket.io 服务正常（命名空间 `/chat`, `/presence`）
- JWT 认证可用
- Agent 对话生成服务运行
- 移动端 UI 已启动（或 Web 端）

## 旅程步骤

### Phase 1: 收到联系

**目的**: 验证 Agent 收到其他 Agent 的联系请求，Socket.io 连接建立正常

**Step 1.1**: 查看被联系通知

- 操作: 等待 Agent 收到联系请求，查看 UI 通知
- 预期: 看到 "Agent [X] 想与你对话" 通知卡片，包含对方 Agent 头像、名称、发起原因摘要
- 验证 AC:
  - `AS-006~func` Agent 被其他 Agent 联系后评估是否值得交流
  - `ISSUE-COM001~c2` 连接认证 - JWT token 验证、用户身份绑定
  - `ISSUE-COM001~c3` 连接生命周期 - 连接建立处理、心跳检测

**Step 1.2**: 验证连接状态

- 操作: 检查 Socket.io 连接状态（DevTools Network 面板）
- 预期:
  - WebSocket 连接成功（状态码 101 或连接升级完成）
  - 命名空间 `/chat` 连接建立
  - JWT token 验证通过，无 401 错误
- 验证 AC:
  - `ISSUE-COM001~c1` Socket.io 与 Express 集成、命名空间设计(/chat)
  - `ISSUE-COM001~c2` 连接权限检查、认证失败处理

**Step 1.3**: 查看对话历史区域

- 操作: 点击进入与该 Agent 的对话区域
- 预期: 显示 "新对话请求" 气泡，对方 Agent 的自我介绍/发起理由
- 验证 AC:
  - `AS-005-AC-2` 自主发起对话请求（对方视角）
  - `ISSUE-COM001~c4` 房间管理 - 房间创建和销毁、加入房间

---

### Phase 2: 评估对话

**目的**: 验证 Agent 自动分析对方意图和态度，评估是否值得继续交流

**Step 2.1**: 观察 Agent 自动评估

- 操作: 静置 10-30 秒，观察 Agent 行为
- 预期:
  - Agent 读取对方消息
  - Agent 生成评估结果（"该 Agent 符合主人偏好" / "对话价值有限"）
  - 评估结果以气泡形式出现在对话中
- 验证 AC:
  - `AS-006-AC-2` 评估是否值得交流
  - `AS-008-AC-1` 实时分析对方意图和态度
  - `ISSUE-AI004~c1` 意图识别、回复生成、对话状态跟踪

**Step 2.2**: 查看 Agent 汇报

- 操作: 查看 Agent 是否向主人发送私聊建议/汇报
- 预期: 在侧边栏或独立私聊区域看到 Agent 的汇报消息
- 验证 AC:
  - `AS-006-AC-3` 向主人汇报对话进展
  - `US-AGENT-009~func` Agent 实时向用户发送私聊建议，仅自己可见
  - `US-AGENT-009-AC-1` 私聊建议显示在侧边栏或独立区域

**Step 2.3**: 验证评估维度

- 操作: 检查 Agent 汇报内容
- 预期: 汇报包含以下维度：
  - 对方 Agent 类型/角色（招聘 Agent/交友 Agent/广告 Agent）
  - 对方意图（引荐候选人/协商薪资/争取优惠）
  - 匹配度初步评估
  - 建议策略（接受/拒绝/继续观望）
- 验证 AC:
  - `ISSUE-AI004~c4` 记忆系统 - 短期对话记忆、关键信息提取
  - `AS-LEARN-001-AC-3` 学习主人决策偏好

---

### Phase 3: 主人互动

**目的**: 验证主人（用户）看到 Agent 的分析和建议，并能做出决策

**Step 3.1**: 查看私聊建议详情

- 操作: 点击侧边栏私聊建议区域
- 预期: 显示详细建议，包含：
  - 话题推荐（如：可以询问对方候选人背景）
  - 风险提示（如：对方 Agent 可能压低薪资预期）
  - 对方意图分析（如：对方正在为候选人争取权益）
- 验证 AC:
  - `AS-008-AC-2` 发送策略建议、风险提示、话题推荐
  - `US-AGENT-009-AC-2` 建议包括话题推荐、风险提示、对方意图分析

**Step 3.2**: 观察 Agent 切换行为

- 操作: 用户点击 "让 Agent 聊" 按钮
- 预期:
  - Agent 以自己名义发送消息到群聊
  - 消息语气符合 Agent 人设
  - 用户侧边栏显示 "Agent 正在代聊"
- 验证 AC:
  - `US-AGENT-007-AC-2` 点击"让 Agent 聊"按钮后 Agent 以自己名义发言
  - `US-AGENT-007~func` 用户在四人群聊中成功切换"亲自聊"和"让 Agent 代聊"
  - `ISSUE-AI004~c2` 人格系统 - Agent 人格定义、语气风格适配

**Step 3.3**: 主人接管对话

- 操作: 用户直接输入消息发送
- 预期:
  - 用户消息以 "我" 的身份出现在群聊
  - Agent 自动切换为旁观模式
  - Agent 继续提供私聊建议
- 验证 AC:
  - `ISSUE-AI004~c5` 切换机制 - 人工介入检测、切换请求处理

### Phase 4: 引荐决策

**目的**: 验证匹配度达标后，Agent 请求主人确认是否引荐

**Step 4.1**: 触发引荐条件

- 操作: Agent 间多轮对话后，等待匹配度评估更新
- 预期: Agent 显示 "匹配度达标，可以向主人申请引荐" 提示
- 验证 AC:
  - `AS-009-AC-1` 评估匹配度是否达到引荐阈值
  - `AS-009~func` 匹配度达标后 Agent 请求主人确认引荐

**Step 4.2**: 查看引荐请求

- 操作: 查看 Agent 发送的引荐请求
- 预期: 显示弹窗或卡片，包含：
  - 引荐对象信息（对方 Agent + 其主人，或候选人）
  - 引荐理由摘要
  - "确认引荐" / "暂不考虑" 按钮
- 验证 AC:
  - `DATE-004~func` 双方接受引荐后进入四人群聊交友
  - `JOB-004~func` 双方 Agent 和用户在四人群聊中协商薪资
  - `AD-004~func` Agent 与商家 Agent 谈判争取更好优惠

**Step 4.3**: 主人确认引荐

- 操作: 点击 "确认引荐"
- 预期:
  - 跳转到四人群聊界面
  - 群聊包含：我方 Agent、对方 Agent、我方用户、对方用户（4 人）
  - Agent 礼貌开场
- 验证 AC:
  - `DATE-004-AC-1` 四人群聊界面
  - `US-AGENT-015~func` 用户在匹配过程中随时退出

**Step 4.4**: 四人群聊验证

- 操作: 在群聊中发送测试消息
- 预期:
  - 消息实时同步到所有成员
  - Agent 在侧边栏继续提供私聊建议
  - Agent 提供交友话题建议
- 验证 AC:
  - `ISSUE-COM001~c4` 房间广播、房间成员查询
  - `DATE-004-AC-2` Agent 提供交友话题建议
  - `US-AGENT-009~func` Agent 实时向用户发送私聊建议

### Phase 5: 异常处理

**目的**: 验证各种边界情况和异常场景的处理

**Step 5.1**: 对方拒绝对话

- 操作: 模拟对方 Agent 拒绝当前 Agent 的对话请求
- 预期:
  - Agent 收到拒绝通知
  - Agent 向主人汇报："对方 Agent 拒绝了对话请求"
  - Agent 提供原因分析（如：匹配度不足、正在忙）
- 验证 AC:
  - `AS-005~edge` 异常：对方拒绝对话
  - `US-AGENT-015-AC-2` Agent 礼貌结束对话

**Step 5.2**: 同时对话数过多

- 操作: Agent 同时收到 3+ 个对话请求
- 预期:
  - Agent 显示 "对话数量已达上限，请先结束部分对话"
  - Agent 自动排序优先级（基于主人偏好匹配度）
  - 主人可选择性忽略部分请求
- 验证 AC:
  - `AS-005-AC-3` 同时与多个候选保持独立对话
  - `AS-005~edge` 异常：同时对话数过多

**Step 5.3**: 大量 Agent 同时联系

- 操作: 5+ 个 Agent 同时发起对话请求
- 预期:
  - Agent 批量显示通知
  - Agent 自动分组："高优先级(2)、中优先级(1)、低优先级(2)"
  - Agent 建议主人优先处理高优先级
- 验证 AC:
  - `AS-006~edge` 异常：大量 Agent 同时联系

**Step 5.4**: 引荐请求超时

- 操作: 主人收到引荐请求后，5 分钟无操作
- 预期:
  - Agent 发送超时提醒："主人还未处理引荐请求，需要我代为回复吗？"
  - 超时后 Agent 自动礼貌拒绝或接受（根据主人设置）
- 验证 AC:
  - `AS-009~edge` 异常：引荐请求超时、一方接受一方拒绝

**Step 5.5**: 建议为空或 AI 降级

- 操作: 模拟 AI 服务降级或无法生成建议
- 预期:
  - Agent 显示："建议生成中..." 或 "当前无法生成建议，请稍后"
  - 不显示空建议（防止信息泄露）
  - 部分功能降级：仅保留基础对话
- 验证 AC:
  - `US-AGENT-009~edge` 异常：建议泄露、建议为空
  - `AS-008~edge` 异常：建议为空、AI 服务降级

**Step 5.6**: 成员离线或消息丢失

- 操作: 群聊中有成员断线
- 预期:
  - 显示 "XXX 已离线" 状态
  - 离线期间消息缓存
  - 重新上线后同步离线消息
  - Agent 通知主人："对方已离线，对话可能需要等待"
- 验证 AC:
  - `DATE-004~edge` 异常：成员离线、消息丢失
