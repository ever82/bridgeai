# BridgeAI 核心概念定义

## Agent 层概念

### CONCEPT-A001: BridgeAI Agent
用户的 AI 代表，基于用户画像和偏好配置，代替用户与其他 Agent 进行初步沟通、筛选和匹配。

### CONCEPT-A002: Agent 画像 (Agent Profile)
用户的数字化描述，分为两个维度：

#### 需求信息 (Demand Profile)
用户在特定场景下想要什么：
- **公开信息**: 可被对方 Agent 浏览的基础需求（脱敏）
- **详细信息**: 仅自家 Agent 可读的具体需求
- **披露控制**: 自家 Agent 决定何时披露哪些细节

#### 供给信息 (Supply Profile)
用户在特定场景下能提供什么：
- **公开信息**: 可被对方 Agent 浏览的基础供给能力（脱敏）
- **详细信息**: 仅自家 Agent 可读的具体供给能力
- **披露控制**: 自家 Agent 决定何时展示哪些能力

#### 通用属性
- 基础属性：昵称、头像、年龄、性别、地理位置
- 场景偏好：VisionShare、AgentDate、AgentJob、AgentAd 的开关
- 沟通风格：直接/委婉、详细/简洁、主动/被动
- 隐私设置：各级信息的披露权限

### CONCEPT-A003: 需求-供给 API (Demand-Supply API)
Agent 之间通过 API 浏览对方需求和供给信息的机制：

#### API 层级
1. **公开层 (Public API)**: 对方 Agent 可浏览的基础信息
   - 需求概要：想要什么类型的东西
   - 供给概要：能提供什么类型的东西
   - 信用评分：可信度参考
   
2. **详情层 (Private API)**: 仅自家 Agent 可读的详细信息
   - 具体需求细节
   - 具体供给能力
   - 敏感个人信息
   - 底价/底线信息

#### 信息披露流程
```
对方 Agent 浏览公开 API
      ↓
判断是否值得交流（匹配度评估）
      ↓
开始 Agent 间对话
      ↓
根据对话进展，自家 Agent 决定：
  - 披露更多需求细节
  - 展示更多供给能力
  - 保持神秘感继续试探
  - 结束对话（不匹配）
```

### CONCEPT-A004: Agent 过滤规则 (Agent Filter Rules)
用户配置的 Agent 匹配过滤条件，控制 Agent 与哪些其他 Agent 进行交流。

#### 地域过滤 (Geo Filters)
- **同城过滤**: 只与同一城市的 Agent 交流
- **同 WiFi 过滤**: 只与同一 WiFi 网络下的 Agent 交流
- **距离过滤**: 只与 N 米/公里范围内的 Agent 交流
- **行政区域**: 限制到省/市/区县级别

#### 属性过滤 (Attribute Filters)
- **基础属性**: 年龄范围、性别
- **生活属性**: 婚恋状态（已婚/未婚/离异）、是否有子女
- **经济属性**: 收入范围、财产状况（房产、车辆等）
- **兴趣属性**: 兴趣爱好标签匹配度
- **社交属性**: 共同好友数、社交圈重叠度

### CONCEPT-A005: 场景专属需求-供给定义

各场景的需求和供给信息定义：

#### VisionShare 场景
**需求信息 (Demand)**:
- **公开**: 想看某地什么情况、大致位置
- **详细**: 具体地点坐标、期望拍摄角度、时间要求、预算上限

**供给信息 (Supply)**:
- **公开**: 是否能去某地拍照
- **详细**: 当前确切位置、设备类型、可拍摄时间、历史照片质量

#### AgentDate 场景
**需求信息 (Demand)**:
- **公开**: 想找什么样的朋友、兴趣爱好
- **详细**: 具体年龄要求、收入期望、房车要求、约会偏好

**供给信息 (Supply)**:
- **公开**: 自己的基本类型、兴趣标签
- **详细**: 真实年龄、收入情况、房产车辆、约会经验

#### AgentJob 场景
**需求信息 (Demand)** - 求职者:
- **公开**: 求职方向、技能领域
- **详细**: 薪资底线、公司规模偏好、远程政策要求

**供给信息 (Supply)** - 求职者:
- **公开**: 技能栈、工作年限
- **详细**: 过往项目细节、薪资期望、可到岗时间

**需求信息 (Demand)** - 招聘方:
- **公开**: 招聘什么岗位、技能要求
- **详细**: 薪资预算上限、团队氛围、发展空间

**供给信息 (Supply)** - 招聘方:
- **公开**: 公司类型、规模、行业
- **详细**: 具体薪资范围、福利待遇、工作强度

#### AgentAd 场景
**需求信息 (Demand)**:
- **公开**: 想买什么品类、预算区间
- **详细**: 品牌偏好、急需程度、最低价期望

**供给信息 (Supply)**:
- **公开**: 有什么商品、大概价格
- **详细**: 成本价、最大优惠空间、库存情况

每个场景有专属的过滤规则配置：

**VisionShare 场景过滤**:
- 距离范围: 照片 GPS 与需求地点距离（如 < 1000 米）
- 时间范围: 照片拍摄时间（如 < 30 分钟）
- 信用要求: 发布者/响应者信用分阈值
- 内容匹配: AI 分析照片是否符合需求场景

**AgentDate 场景过滤**:
- 基础属性: 年龄范围、性别
- 生活属性: 婚恋状态（单身/已婚/离异）、是否有子女
- 地域属性: 同城、距离范围（如 < 50 公里）
- 经济属性: 收入范围、房产/车辆状况
- 兴趣匹配: 共同兴趣标签、匹配度阈值
- 信用要求: 最低信用分、无不良记录

**AgentJob 场景过滤**:
- 技能匹配: 技能标签匹配度、经验年限
- 薪资范围: 求职者期望与职位预算重叠度
- 工作地点: 城市、远程政策（远程/混合/现场）
- 公司规模: 初创/中型/大厂偏好
- 行业领域: 特定行业筛选
- 信用要求: 求职者/招聘方信用分

**AgentAd 场景过滤**:
- 需求匹配: 商品/服务类别匹配
- 预算范围: 用户预算与商品价格匹配
- 品牌偏好: 白名单/黑名单品牌
- 地理位置: 同城/可配送范围
- 商家信用: 最低信用分、好评率
- 广告频次: 每日接收广告数量上限

### CONCEPT-A006: Agent 通信协议 (Agent Message Protocol)
标准化的 Agent 间通信格式，包含：
- 协议版本
- 发送/接收 Agent ID
- 场景类型
- 消息类型（请求/响应/协商/确认/拒绝）
- 有效载荷（意图、约束、报价、询问）
- **需求-供给信息**（通过 API 交换）
- 信任评分和时间戳
- **通信者身份标识**（Agent / Human）

### CONCEPT-A007: 信息披露控制 (Information Disclosure Control)
自家 Agent 根据对话情况决定向对方披露哪些信息的机制：

#### 披露策略
- **渐进披露**: 随着信任建立逐步透露更多信息
- **条件披露**: 对方满足某些条件后才披露（如先透露自己的）
- **交换披露**: 以信息换信息（"我可以告诉你，但你也得告诉我 X"）
- **底线保护**: 底价、底线信息绝不轻易透露

#### 披露等级
- **L1 - 公开**: 任何人可见
- **L2 - 初步**: 对方 Agent 可见（公开 API）
- **L3 - 深入**: 匹配后可见
- **L4 - 私密**: 只有对方用户可见
- **L5 - 绝密**: 只有自家 Agent 知道（Private API），绝不透露

### CONCEPT-A008: Agent 人机切换 (Agent-Human Handoff)
在四人群聊中，用户随时切换"亲自聊"或"让 Agent 聊"：

#### 切换触发
- 用户点击切换按钮
- Agent 遇到复杂情况请求切换
- 预设关键字触发

#### 切换感知
- 群内显示"甲用户已亲自上线"
- 消息头像/名称实时变化
- 输入框模式同步更新

### CONCEPT-A009: Agent 匹配引擎 (Agent Matching Engine)
计算两个 Agent 匹配度的算法，基于：
- **需求-供给匹配度**（双方需求与供给的契合程度）
- 意图匹配度
- 约束条件兼容性
- 地理位置邻近度
- 属性过滤条件满足度
- 历史交互信任度
- **过滤规则兼容性**（双方过滤条件是否互斥）
- 输出：匹配度评分（0-100）和过滤通过状态

### CONCEPT-A010: Agent 对话 (Agent Dialogue)
两个 Agent 之间的多轮自动对话过程，采用"四人群聊"模式：
- **群聊成员**: 甲用户、甲 Agent、乙用户、乙 Agent
- **可见性**: Agent 间的对话群内所有人可见
- **私聊建议**: Agent 给主人的建议仅主人可见
- **身份标识**: 每条消息清晰标识发送者（用户/Agent）

### CONCEPT-A011: Agent 私聊建议 (Agent Whisper)
Agent 在群聊过程中给主人的私下建议，仅主人可见：
- 对话策略建议
- 对方意图分析
- 风险提示
- 话题推荐

### CONCEPT-A013: 智能需求提炼 (Smart Demand Extraction)
用户通过自然语言与 Agent 对话，Agent 自动提炼结构化需求：

#### 提炼流程
```
用户: "我想看看三里屯现在堵不堵，最好有照片"
   ↓
Agent 分析:
  - 场景: VisionShare
  - 地点: 三里屯 (需解析精确坐标)
  - 内容: 交通路况
  - 形式: 照片
  - 时效: 现在 (即时)
   ↓
生成需求记录
```

#### 需求记录结构
```typescript
interface DemandRecord {
  id: string;
  scene: string;
  
  // 基础属性（系统可快速查询）
  queryable: {
    location: { lat: number; lng: number; radius: number };
    timeRange: { start: Date; end: Date };  // 时效性
    category: string;  // 需求类别
    budget?: { min: number; max: number };
  };
  
  // 公开条件（对方 Agent 智能阅读理解）
  readable: {
    description: string;  // 自然语言描述
    requirements: string[];  // 具体要求列表
    preferences: string[];   // 偏好列表
  };
  
  // 私有条件（仅自家 Agent 掌握）
  private: {
    mustHaves: string[];     // 硬性要求
    dealBreakers: string[];  // 绝对不接受的条件
    bottomLine: any;         // 底线
    urgency: 'critical' | 'high' | 'medium' | 'low';
  };
  
  status: 'active' | 'matched' | 'fulfilled' | 'expired';
  createdAt: Date;
  expiresAt: Date;  // 自动过期时间
}
```

### CONCEPT-A014: 智能供给提炼 (Smart Supply Extraction)
类似需求提炼，Agent 从用户对话中提取供给能力：

```typescript
interface SupplyRecord {
  id: string;
  scene: string;
  
  // 基础属性（系统可快速查询）
  queryable: {
    location: { lat: number; lng: number; radius: number };
    availableTime: { start: Date; end: Date };  // 可服务时间
    category: string;
    priceRange?: { min: number; max: number };
  };
  
  // 公开条件（对方 Agent 可查阅）
  readable: {
    capabilities: string[];  // 能力列表
    features: string[];      // 特点
    conditions: string[];    // 提供条件
  };
  
  // 私有条件（仅自家 Agent 掌握）
  private: {
    actualCapacity: number;  // 实际承载能力
    costStructure: any;      // 成本结构
    minimumPrice: number;    // 最低价
    constraints: string[];   // 限制条件
  };
  
  status: 'available' | 'matched' | 'consumed' | 'expired';
}
```

### CONCEPT-A015: 三级匹配过滤算法 (Three-Layer Matching Algorithm)

#### 第一层：系统实时查询匹配（一对多）
```
系统收到新需求/供给 → 立即从数据库查询所有匹配项

查询模式:
- 一对多: 一个需求 → 匹配多个供给
- 一对多: 一个供给 → 匹配多个需求

查询维度:
- 地理位置: 在半径范围内 (如 1km / 50km)
- 时间范围: 在有效期内 (需求未过期 / 供给仍可用)
- 类别匹配: 场景和类别一致
- 硬性过滤: 信用分、基础属性等

输出: 候选匹配列表 (可能有几十个到几百个)
示例: "三里屯拍照需求" → 匹配到 50 个附近的潜在供给方
```

#### 第二层：Agent 智能筛选排序（一对多 → 精选多）
```
自家 Agent 收到候选列表 → 批量评估排序

评估维度:
- 读取所有候选的公开条件 (readable)
- 对比自家私有条件 (private)
- 智能计算匹配度分数:
  * 地理位置邻近度 (权重 30%)
  * 需求供给契合度 (权重 40%)
  * 对方信用分 (权重 20%)
  * 时效性紧迫度 (权重 10%)

Agent 决策:
- Top N 高匹配度: 同时发起聊天或批量展示给用户
- 中等匹配度: 备用候选，优先匹配失败后启用
- 低匹配度: 忽略

输出: 精选匹配列表 Top 10-20
示例: 从 50 个候选中精选出 10 个最合适的
```

#### 第三层：多对多 Agent 对话核对（四人群聊矩阵）
```
用户可以同时与多个候选进入四人群聊

场景示例:
- VisionShare: 需求方同时与 3-5 个供给方 Agent 聊
  * 群聊1: 需求方 + Agent ↔ 供给方A + Agent
  * 群聊2: 需求方 + Agent ↔ 供给方B + Agent
  * 群聊3: 需求方 + Agent ↔ 供给方C + Agent

- AgentDate: 同时与多个匹配对象聊天筛选
- AgentJob: 同时与多个候选人/公司沟通
- AgentAd: 同时与多个商家比价谈判

决策点:
- 选择最优 → 确认匹配 → 请求主人确认 → 引荐真人
- 全部不合适 → 从备用候选中继续
- 需要补充信息 → 请主人亲自下场
```

### CONCEPT-A016: 实时匹配触发机制 (Real-time Matching Trigger)

#### 主动匹配（新需求触发一对多）
```
用户发布新需求
    ↓
系统立即查询所有匹配的历史供给（一对多）
    ↓
Agent 批量评估排序（Top N）
    ↓
符合条件 → 批量推送通知给用户
    ↓
用户选择：
  - 与多个候选同时开始聊天（多线程）
  - 逐个查看后选择聊天对象
  - 等待更多匹配（系统持续扫描）

示例: 
用户: "想看三里屯路况"
系统: 找到 30 个潜在响应者
Agent: 排序后推荐前 10 个
用户: 同时与 3 个响应者 Agent 进入群聊
```

#### 被动匹配（新供给触发一对多）
```
用户发布新供给
    ↓
系统查询所有匹配的历史需求（一对多）
    ↓
Agent 评估所有需求方的匹配度
    ↓
批量通知需求方 Agent
    ↓
需求方 Agent 选择是否发起聊天
    ↓
多个需求方可能同时与供给方聊天

示例:
用户: "我在三里屯，能拍照"
系统: 找到 15 个匹配的需求
Agent: 筛选出 5 个高匹配需求
结果: 5 个需求方 Agent 同时联系供给方
```

#### 持续匹配（定时扫描增量匹配）
```
系统定时扫描未匹配成功的需求/供给
    ↓
发现新的潜在匹配（如新用户加入、位置更新）
    ↓
Agent 重新评估所有候选
    ↓
推送新的匹配通知
    ↓
用户可选择加入新的聊天
```

### CONCEPT-A017: 多对多聊天矩阵 (Multi-Chat Matrix)
用户可以同时维护多个四人群聊：

```
                    用户A + AgentA
                         |
        ┌────────────────┼────────────────┐
        │                │                │
   群聊1              群聊2              群聊3
        │                │                │
   用户B1+AgentB1   用户B2+AgentB2   用户B3+AgentB3
        
用户A 可以同时与 B1、B2、B3 聊天
每个聊天都是独立的四人群聊
用户A 的 Agent 在每个群中提供建议
用户A 可随时切换到任意群聊亲自沟通
```

### CONCEPT-A017: 四人群聊室 (Quad Chat Room)
BridgeAI 的核心交流场所，一个聊天室内同时存在四个参与者：
- **甲用户 (User A)**: 需求方/求职者/消费者等
- **甲 Agent (Agent A)**: 甲用户的 AI 代表
- **乙用户 (User B)**: 供给方/招聘方/商家等
- **乙 Agent (Agent B)**: 乙用户的 AI 代表

#### 消息可见性
- **群聊消息**: 所有人可见（四方都能看到）
- **私聊建议**: 仅自己可见（只有自己的 Agent 给自己的建议）
- **系统通知**: 所有人可见（如"甲用户已上线"）

#### 身份标识
每条消息清晰标识发送者：
- 头像区分：Agent 用机器人头像，用户用真人头像
- 名称区分："小A (Agent)" / "小A的主人" / "用户昵称"
- 颜色区分：Agent 消息和用户消息用不同颜色背景

## 信用与评分系统

### CONCEPT-R001: 真人评分机制 (Human Rating System)
双方进入真人交流后，可以对对方进行评分，评分影响后续匹配。

#### 评分维度
- **诚信度**: 对方描述是否真实（1-5星）
- **礼貌度**: 交流是否礼貌尊重（1-5星）
- **响应度**: 回复是否及时（1-5星）
- **满意度**: 整体体验（1-5星）
- **场景专属**: 如 VisionShare-照片质量、AgentDate-约会体验

#### 评分使用
- 计算用户综合信用分
- 作为过滤条件（如"只与评分>4星的 Agent 交流"）
- 影响匹配排序（高评分用户优先推荐）
- 低评分用户限制某些功能

### CONCEPT-R002: 信用分计算 (Credit Score Calculation)
```
信用分 = 基础分(60)
       + 完成交易数 × 2
       + 平均评分 × 10
       - 被投诉数 × 5
       - 被举报数 × 10
       + 跨场景信用加成
       
范围: 0-100
等级: 优秀(90+)、良好(75-89)、一般(60-74)、较差(<60)
```

### CONCEPT-R003: 评分过滤规则 (Rating Filter Rules)
用户可设置的基于评分的过滤条件：
- **最低信用分**: 只与信用分 > N 的 Agent 交流
- **最低评分**: 只与平均评分 > N 星的用户交流
- **交易次数**: 只与完成交易 > N 次的用户交流
- **无不良记录**: 只与无投诉/举报记录的用户交流

## 场景层概念

### VisionShare 场景

#### CONCEPT-V001: 视野请求 (VisionRequest)
用户发出的"查看某地实时情况"的请求，包含目标地点、期望看到的内容、紧迫程度、愿意支付的积分。

#### CONCEPT-V002: 视野任务 (VisionTask)
发布给附近用户的具体任务，包含位置坐标、拍摄要求、报酬、有效期。

#### CONCEPT-V003: 视野照片 (VisionPhoto)
用户拍摄或从相册检索的照片，包含拍摄地点、拍摄时间、拍摄者信息、关联的任务ID、脱敏状态。

#### CONCEPT-V004: AI 智能检索本地相册 (AI Local Gallery Scan)
App 在用户已授权的相册范围内自动扫描，通过 GPS 地点匹配、拍摄时间筛选、AI 内容分析，找出符合任务需求的照片。

#### CONCEPT-V005: AI 隐私脱敏 (AI Privacy Sanitization)
照片上传前自动进行的隐私保护处理，包括人脸模糊/打码、去除 EXIF 元数据敏感信息。

### AgentDate 场景

#### CONCEPT-D001: 交友画像 (Dating Profile)
用户在 AgentDate 场景下的专属画像，包括：
- 个人介绍、兴趣爱好
- 期望对象的条件（年龄范围、性别、婚恋状态）
- 约会偏好
- 隐私级别设置
- **交友过滤规则**（只与未婚、收入范围X、年龄范围Y的交流）

#### CONCEPT-D002: 匹配推荐 (Match Recommendation)
Agent 向用户推荐的潜在交友对象，包含：
- 对方脱敏后的基本信息
- 匹配度评分
- 共同兴趣标签
- 匹配理由说明
- **对方信用分和评分记录**

#### CONCEPT-D003: 双向同意 (Mutual Consent)
AgentDate 的核心机制，必须双方用户都同意引荐后，才能开始真人沟通。

### AgentJob 场景

#### CONCEPT-J001: 求职画像 (Job Seeker Profile)
求职者在 AgentJob 场景下的专属画像，包括：
- 技能栈和经验年限
- 期望薪资范围
- 工作偏好（远程/onsite、公司规模、行业）
- 职业规划
- **求职过滤规则**

#### CONCEPT-J002: 职位画像 (Job Position Profile)
招聘方发布的职位信息，包括：
- 技能要求（硬性/软性）
- 薪资预算范围
- 工作条件
- 团队信息
- **招聘过滤规则**

#### CONCEPT-J003: 简历匹配 (Resume Matching)
Agent 间自动进行的求职者与职位匹配评估，输出匹配度评分和过滤通过状态。

### AgentAd 场景

#### CONCEPT-AD001: 消费需求画像 (Consumer Need Profile)
用户在 AgentAd 场景下的消费需求描述，包括：
- 感兴趣的商品/服务类别
- 预算范围
- 品牌偏好/黑名单
- 购买周期
- **消费过滤规则**

#### CONCEPT-AD002: 商家优惠画像 (Merchant Offer Profile)
商家的优惠推广信息，包括：
- 目标客户画像
- 优惠内容和条件
- 推广预算
- 有效期
- **商家信用要求**

#### CONCEPT-AD003: Agent 优惠协商 (Agent Negotiation)
用户 Agent 与商家 Agent 之间的"讨价还价"过程，争取更优惠的条件。

## 基础层概念

### CONCEPT-B001: Bridge 积分 (Bridge Credits)
跨场景通用的平台虚拟货币。

### CONCEPT-B002: 信用分 (Trust Score)
用户信用评分，基于所有场景的综合表现计算。

### CONCEPT-B003: 隐私等级 (Privacy Level)
用户可设置的信息披露分级：
- L1: 仅 Agent 可见（最私密）
- L2: 匹配 Agent 可见（脱敏）
- L3: 引荐后可见（部分真实信息）
- L4: 完全公开
- **L-Agent**: Agent 可见但真人不可见（敏感信息）

### CONCEPT-B004: 场景切换 (Scene Switching)
用户在同一 App 中切换不同场景的能力。

### CONCEPT-B005: WiFi 指纹 (WiFi Fingerprint)
用于同 WiFi 过滤的技术，通过 WiFi SSID/BSSID 识别用户是否在同一网络下。

### CONCEPT-B006: 实时位置 (Real-time Location)
用于距离过滤和同城过滤的实时 GPS 位置信息，支持：
- 静态位置（注册地）
- 实时位置（当前 GPS）
- 位置更新频率控制（隐私保护）
